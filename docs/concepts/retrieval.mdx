Reading files by hand stops scaling somewhere past a few hundred notes. The
retrieval layer gives every compiled vault a local hybrid search index so an
agent can ask a question and get the right chunks back — while the answer to
"what can this search see?" stays exactly the same as everywhere else in the
system: only what the person may read.

## The boundary comes for free

The index for a person is built **only from their compiled vault** — never from
master. The compiler already removed everything they may not read, so the index
physically cannot contain it. There is no query-time permission filter to get
wrong: retrieval inherits the structural boundary.

```mermaid
flowchart LR
  M[master] -->|compile| V["&lt;out&gt;/&lt;person&gt;/"]
  V -->|brain index| I[".brain/index.db"]
  I -->|brain search / MCP| A[agent]
```

This is why there is no "master index filtered per query" option: that would
move enforcement from the compiler (structural) to a filter (a bug away from a
leak). A [randomized leak property test](/concepts/the-compiler) proves no
query for a person ever returns a chunk outside their readable spaces.

## How a query is answered

Search fuses three independent legs by reciprocal-rank fusion (RRF), and needs
no score calibration between them:

- **Keyword** — SQLite FTS5 (BM25) over chunk text.
- **Vector** — semantic nearest-neighbours via [sqlite-vec](https://github.com/asg017/sqlite-vec).
- **Graph** — personalized PageRank over the wikilink and fact-co-mention graph,
  seeded by entity names and aliases matched in the query, the top text hits,
  and optionally a **center note** (`--center`, or the `center` argument of the
  `brain_search` tool — typically the note you're working from). Unlike the two
  text legs, this leg can introduce results of its own that never textually
  match; such hits are attributed to the `graph` source.

Reciprocal-rank fusion means a chunk multiple legs surface outranks one only a
single leg found. Notes are chunked heading-aware (each chunk carries its
`Heading > Subheading` path), code fences are never split, and results are
deduped to at most two chunks per file. If no embedding provider is configured
— or `sqlite-vec` can't load — search degrades to **keyword + graph** with a
warning, never an empty result.

An unknown center note degrades to a plain search (keyword + vector, or
keyword-only if `sqlite-vec` is unavailable) with a warning.

<Callout type="info">
  `sqlite-vec` loads as a SQLite extension, so it needs a Python whose `sqlite3`
  was compiled with loadable-extension support. Homebrew, pyenv, uv-managed, and
  most Linux distro Pythons qualify; the **python.org macOS installer** ships it
  disabled. Verify with
  `python -c "import sqlite3; print(hasattr(sqlite3.Connection, 'enable_load_extension'))"`
  — `False` means the vector leg is unavailable and search runs keyword-only.
</Callout>

## The `.brain/` contract

The index lives at `<vault>/.brain/index.db`. It is **machine-local state**, on
the same footing as `.git`:

- **Never committed.** The compiler emits a `.gitignore` (`.brain/`,
  `.obsidian/`), so it never enters the vault's git history.
- **Rebuilt where it's used.** Employees run `brain index` after they pull;
  hosted agents get theirs from `brain cycle --index` server-side.
- **Preserved across recompiles.** A recompile swaps the vault contents but
  carries `.brain/` across, exactly like `.git`.

## Incremental by design

The compiler does a full rebuild every run, so the indexer diffs the manifest's
per-file sha256 against what the index already holds: unchanged files are
skipped, and a changed file only re-embeds chunks whose content hash misses the
embedding cache. The cache is keyed by `(chunk_sha, model)` and can be shared
across people, so a chunk two vaults have in common embeds exactly once.

## Scale tiers

The vector store sits behind a three-method seam (`add`/`delete`/`knn`), so the
backend swaps without touching chunking, FTS, or fusion. The unit of scale is
one person's vault, not the whole company.

| Corpus (per vault) | Backend | Trade-off |
|---|---|---|
| ≤ 50K pages | sqlite-vec brute-force (default) | Exact KNN, zero infra, single file. Storage grows linearly. |
| 50K–500K pages | + int8 quantization | 4× smaller, near-exact, same file — still linear scan. |
| millions | LanceDB IVF-PQ or sqlite-vec DiskANN | Sublinear ANN; a heavier dependency or a pre-release. |

Ship tier 1; reach for the others only when a real vault crosses the line.

## Facts and time

Search finds relevant text; some questions instead need a queryable state —
"is this client still active", "who's the current contact" — answered as of
a date rather than by relevance. Those are written as fact lines: a bullet
with a `[from::]` date and a `[source::]` back to the note that established
it, e.g.

```markdown
- Acme is a client [from:: 2025-03] [source:: [[2025-03-12 Kickoff call]]]
- Acme's plan is Enterprise [from:: 2025-03] [until:: 2026-01] [source:: [[2025-03-12 Kickoff call]]]
- Acme's plan is Growth [from:: 2026-01] [source:: [[2026-01-08 Downgrade]]]
```

A fact is never edited or deleted once superseded — the old line gets an
`[until::]` and a new line is added, so the full history stays on the record.
Three questions read that history differently:

- **Current** (default) — facts true today.
- **As-of a date** (`--as-of`) — facts true on that date, using what the
  vault knows now. Answers "what was true then," with the benefit of
  hindsight.
- **Believed-on a date** (`--believed-on`) — what the vault would have told
  you if asked on that date, replayed from the git history at that point in
  time. Answers "what did we think then," including facts since corrected.

That last distinction is why **transaction time is git**: valid time (when a
fact was true) lives in the `[from::]`/`[until::]` dates in the text, while
transaction time (when the vault learned it) is just which commit you read
the file at. No separate audit log is needed — git already keeps one.

This is a different job from the `[source](URL), as of YYYY-MM` citations
used elsewhere: citations back narrative prose ("Acme relaunched their
site"), while fact lines are for state a query can ask about later. Use a
citation for a one-off observation; use a fact line for anything future
questions will need to look up.

Query facts with [`brain facts`](/reference/cli#brain-facts), or the
`brain_facts` MCP tool. The dashboard shows them too: a Facts tab in the live dashboard, and a Facts section in the static snapshot — each with a date control for what was true on a given day.

## Using it from an agent

Register the MCP server once and any MCP client (Claude Code, Cursor, Codex,
Claude Desktop) can search the vault as a tool:

```bash
claude mcp add brain -- brain mcp --vault ~/brain
```

It exposes five read-only tools: `brain_search` (hybrid search with keyword,
vector, and graph legs; optional center note as a strong seed for graph-proximity
ranking), `brain_read` (a path-scoped
file read that refuses symlinks, vault escapes, and anything outside a
readable space), `brain_links` (a note's backlinks and outgoing wikilinks),
`brain_recent` (recently changed notes, from the vault's git history), and
`brain_facts` (time-stamped fact lines — current, as-of, or believed-on a
date). See the [CLI reference](/reference/cli) for `brain index`,
`brain search`, `brain facts`, and `brain mcp`.
