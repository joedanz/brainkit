Company Brain · Deployment Plate №1

Where the brain lives

The decided architecture: two boxes. Employees are non-technical and chat is their entire product — they message their own agent on WhatsApp or Telegram and never see git, files, or the word "vault." All agents run server-side, one container per person, on the agents box; the master lives alone on the brain box.

In plain English Every employee texts their own private AI assistant, the way they'd text a person. The assistant keeps their notes in a private notebook only they can see. A separate "librarian" computer holds the company's shared knowledge: when someone chooses to share something, a human approves it, and it then appears in everyone's notebook. Nothing is shared by accident, because each assistant is physically given only its own person's notebook — there is no setting to get wrong.

1–2Chat is the whole product

"Remember X" → the agent writes it into Inbox/ in its own clone. Questions → local search over the compiled vault. "Share this with the team" → the agent drafts a promotion. Users never touch git, Obsidian, or files; the pairing-code flow is the entire IT setup they ever experience.

2One container per person

Isolation is structural: a container cannot read another person's vault because it was never mounted — the only defense that survives prompt injection. Cost is ≈300 MB RAM each, idle CPU near zero: ~5 people → 4 GB · ~15 → 8 GB · ~30 → 16 GB. Adding an employee is one compose stanza.

3–5Two boxes, one boundary

The master's isolation is hardware, not configuration — there is no compose line that can drift into exposing it. If the agents box is ever compromised: rotate deploy keys, rebuild containers, the master's history is intact. If the brain box is down, agents keep working on local clones and sync drains later.

Company Brain · Deployment Plate №2

How agents route back

An agent never addresses the master. It addresses its own vault's git remote — which happens to live on the brain box. Everything an agent wants to tell the company is a commit; the master extracts meaning from pushes on its own schedule, asynchronously and fail-closed.

2One key, one repo

Agents hold zero credentials to the brain system itself — no API tokens, no service accounts. The only secret in any container is one SSH key that reaches one compiled vault. Compromise an agent and the blast radius is: push garbage to its own vault (reverted by validation) and read what its person could already read.

4Never in the hot path

The brain box needs no availability SLA for agents to function. If it's down, every agent keeps chatting, reading, and capturing against its local clone; commits queue in local git history and drain when it returns. The master is a batch consolidator, not a dependency any agent waits on.

ATransport is swappable

We chose Tailscale (A): both boxes join one tailnet, the brain box exposes no public ports, and the same net carries your admin SSH and the dashboard tunnel. If agents ever run outside the tailnet, add B for just those; move to a forge (C) only if you'd rather not operate SSH at all.

Company Brain · Deployment Plate №3

The visual layer — seeing your agent

Chat stays the primary product, but every container can also expose visual surfaces. Because sessions live in the container's shared state.db, a conversation started on WhatsApp continues in the browser or the desktop app — same brain, different doors. The one rule: the auth gate must map identity to container, so Alice can only ever reach Alice's backend.

✓Yes, the desktop app attaches

Hermes Desktop has first-class remote-backend support: the container runs hermes serve --host … --port 9119 (the same headless backend the app uses locally), a non-loopback bind auto-engages the auth gate, and Alice enters her URL under Settings → Gateway and signs in. The app runs as a GUI-only "lite" client — no local agent installed on her laptop.

1The brain dashboard is the sleeper

We already ship the right first visual surface: the vault's own live dashboard — employee-first, read-only, typographic, built to show what's new, what links where, and what needs attention. It answers "what does my assistant actually know?" without handing anyone a control panel, which is exactly the right amount of power for a non-technical org's first step.

!Auth is the real decision

Exposing hermes serve means remote control of that agent, so credentials must be per-person, per-container — never shared. Pilot: keep all visual surfaces tailnet-only. Wider rollout: one SSO reverse proxy (OIDC / Nous OAuth) mapping each login to exactly one container's ports. Basic-auth on the public internet is explicitly discouraged by the hermes docs.

Company Brain · Deployment Plate №4

The deployment runbook

Everything to stand this up, in order. Phases 1–3 happen once and take an afternoon. Phase 4 is the ~10 minutes you spend per new employee. Phase 5 is the steady state — a few minutes a week.

1Provision — once

  1. Two VPSes, same provider & region. Brain box: 2 GB is plenty. Agents box: ≈300–400 MB per employee plus 2 GB headroom (15 people ≈ 8 GB).
  2. Ubuntu LTS, key-only SSH, unattended-upgrades on both.
  3. Join both to one tailnet; no other inbound ports anywhere.
# on both boxes
curl -fsSL https://tailscale.com/install.sh | sh
tailscale up   # names: brain-box, agents-box

2Brain box — once

uv tool install brainkit
brain init /srv/brain/master --company "Acme Co"
$EDITOR /srv/brain/master/_meta/org.yaml     # people, roles, teams
$EDITOR /srv/brain/master/_meta/spaces.yaml  # visibility rules
brain compile --master /srv/brain/master --out /srv/brain/compiled
brain doctor  --master /srv/brain/master --out /srv/brain/compiled

# one sync user; per-person keys locked to one repo each
adduser --disabled-password brain-sync
# crontab — the whole orchestration layer
*/5 * * * * brain cycle --master /srv/brain/master \
  --out /srv/brain/compiled --index --json \
  >> /var/log/brain/cycle.jsonl || curl -fsS $ALERT_URL
17 3 * * *  brain doctor --master /srv/brain/master \
  --out /srv/brain/compiled || curl -fsS $ALERT_URL
23 4 * * *  git -C /srv/brain/master push --mirror offsite

The $ALERT_URL is a healthchecks.io-style ping — a nonzero cycle means a rejected write-back; a nonzero doctor means an integrity error. The nightly mirror push to one private offsite remote backs up the master. Optional: set BRAIN_EMBED_BASE_URL (an embedding API reachable from both boxes) to upgrade keyword search to semantic — cycle --index and each container's brain index pick it up from the environment.

3Agents box — once

  1. Install Docker + the compose plugin.
  2. Build one hermes image with brainkit baked in (agents need brain index / search / mcp locally).
  3. Set an LLM provider key once, shared via compose env.
# docker-compose.yml — one stanza per person
services:
  agent-alice:
    image: hermes-brain:latest
    restart: unless-stopped
    environment:
      - TELEGRAM_BOT_TOKEN=${ALICE_BOT_TOKEN}
      - ANTHROPIC_API_KEY=${LLM_KEY}
      - BRAIN_EMBED_BASE_URL=${EMBED_URL}  # optional: semantic search
    volumes:
      - alice-state:/root/.hermes   # memory, sessions, pairing
      - alice-vault:/vault          # her compiled vault clone

Each container's first boot: install the company-brain hermes profile (SOUL.md, terminal.cwd: /vault, brain skills), clone brain-sync@brain-box:/srv/brain/compiled/<person> with that container's deploy key, run brain index, and start the profile's cron: pull → index → push every few minutes.

4Onboard an employee — ~10 min each

  1. Add them to org.yaml; the next cycle compiles their vault.
  2. Brain box: generate their deploy key, append one authorized_keys line forced to their repo.
  3. Telegram: @BotFather → /newbot → copy the token (WhatsApp: link a number via the hermes WhatsApp adapter).
  4. Agents box: copy-paste a compose stanza, set the token, docker compose up -d agent-<name>.
  5. They message their bot; you approve the one-time pairing code.
  6. Send them one line: "This is your assistant — tell it things, ask it things." That's their entire onboarding.

5Operate — minutes per week

  1. Approvals: ssh brain-box brain promotions list / approve <id> --approver joe; review in the dashboard over ssh -L on the tailnet.
  2. Watch two numbers: cycle exit ≠ 0 (rejected write-back) and doctor exit ≠ 0 (integrity) — both already ping your alert URL.
  3. Updates: uv tool upgrade brainkit on the brain box; rebuild the hermes image and docker compose up -d to roll agents.
  4. Backups, both halves: the master mirror covers knowledge; the agents box volumes hold what git doesn't — agent memories, sessions, and pairing state. Snapshot the volumes (or the whole VPS) nightly.
  5. Offboarding: remove from org.yaml, delete their authorized_keys line, docker compose rm their container.

6Visual layer — optional, phased

  1. Bake the extras into the image: .[web,pty] for hermes serve; brainkit is already there for the brain dashboard.
  2. Pilot (tailnet-only): expose each container's two ports on the agents box's tailscale IP; work devices join the tailnet.
  3. Wider rollout: one SSO reverse proxy (Caddy/Traefik + OIDC or Nous OAuth via hermes dashboard register) with per-person routes.
# inside each container (supervised alongside the gateway)
hermes serve --host 0.0.0.0 --port 9119   # desktop + web backend
brain dashboard --vault /vault --host 0.0.0.0 --port 8787

# proxy routes — identity must map to container
alice.agents.example.com       → agent-alice:9119
alice-brain.agents.example.com → agent-alice:8787

Per-person credentials in each container's .env (HERMES_DASHBOARD_BASIC_AUTH_* on the tailnet; OAuth beyond it). The brain dashboard has no auth of its own — off-loopback it must sit behind the authenticated proxy, never exposed directly.

✓Failure drills — reassuringly boring

  1. Brain box down: agents keep chatting and capturing on local clones; commits queue and drain on recovery. Users notice nothing.
  2. Agents box down: chat goes quiet, zero data lost; docker compose up -d restores everything from volumes.
  3. Agents box compromised: rotate deploy keys, rebuild containers from the image — the master's history is intact and trustworthy.
  4. Brain box lost: restore the mirror, recompile, re-point remotes. The offsite mirror is the whole plan.
The rule of thumb: the brain box is plumbing with a permanent address; every individual brain travels with its agent on the agents box; and the boundary between them is hardware, not configuration — the cheapest security control this system will ever buy.
