This guide is the end-to-end worked example of the [deployment rule's](/guides/deployment) container branch: every personal agent runs server-side, in a container that mounts only that person's compiled vault. It fits teams where employees are **non-technical and chat is the entire product** — they message their own agent on WhatsApp or Telegram and never see git, files, or the word "vault."

<Callout type="info">
  **In plain English:** every employee texts their own private AI assistant. The
  assistant keeps their notes in a private notebook only they can see. A separate
  "librarian" machine holds the company's shared knowledge: when someone chooses to
  share something, a human approves it, and it then appears in everyone's notebook.
  Nothing is shared by accident, because each assistant is physically given only its
  own person's notebook — there is no setting to get wrong.
</Callout>

## The shape

Two small hosts joined by one private git link. [Several transports work](#choosing-the-transport) — pick by situation; with a private network or a tailnet, there are no public ports anywhere.

```text
Employees ── WhatsApp / Telegram ──┐  (chat is the only user surface)
                                   ▼
┌─ AGENTS BOX — one VPS, Docker ─────────────────────────────┐
│  ┌─ container: alice ─┐ ┌─ container: bob ─┐ ┌─ carol… ─┐  │
│  │ hermes gateway     │ │ hermes gateway   │ │          │  │
│  │ + agent            │ │ + agent          │ │          │  │
│  │ /vault = alice/    │ │ /vault = bob/    │ │          │  │
│  │ .brain index + MCP │ │ .brain index+MCP │ │          │  │
│  └────────────────────┘ └──────────────────┘ └──────────┘  │
└───────────────▲─────────────────▲──────────────────────────┘
                │ git over SSH — the only link between the boxes,
                │ per-person deploy keys, one key → one repo
                ▼
┌─ BRAIN BOX — master host, AGENT-FREE ──────────────────────┐
│  master vault ─→ brain cycle (cron) ─→ compiled/<person>   │
│  promotion queue · brain doctor · localhost dashboard      │
└─────────────────────────────────────────────────────────────┘
```

<CardGroup cols={2}>
  <Card title="Brain box — plumbing" icon="server">
    git + cron + the `brain` CLI, nothing else. **No LLM ever runs here**, so
    nothing here can be talked into leaking. 2 GB of RAM is plenty.
  </Card>
  <Card title="Agents box — one container per person" icon="box">
    Each container holds a complete personal world: hermes gateway + agent, its
    own bot token and memory, and a clone of **only** that person's compiled
    vault. ≈300 MB each; LLM compute happens at the API, not here.
  </Card>
</CardGroup>

**Why two boxes instead of one?** The agents box runs LLM-driven processes with chat-reachable attack surface; the brain box holds every person's private space. Separate hardware makes the boundary structural — there is no compose line that can drift into exposing the master. The entire cost of that guarantee is one extra $5–10/month VPS.

## How the sync works

An agent never addresses the master. It addresses *its own vault's git remote*, which happens to live on the brain box:

1. The agent edits its own spaces in its local clone and commits — captures go to its `Inbox/`, share drafts to `People/<me>/Promotions/`. This works even when the brain box is down.
2. `git push` — the entire client interaction. Pushes are **accepted as-is**; validation is asynchronous.
3. [`brain cycle`](/reference/cli#brain-cycle) (cron, every few minutes) validates every pushed path server-side, applies what's legal, reverts the rest on the next compile (fail closed), sweeps promotion drafts into the [pending queue](/concepts/promotions), and recompiles every vault.
4. `git pull` — the verdict returns: rejected edits vanish, approved shared knowledge arrives.

<Callout type="info">
  **Never in the hot path.** Agents hold zero credentials to the brain system —
  the only secret in a container is one SSH key that reaches one repo. And the
  brain box needs no availability SLA: if it's down, agents keep chatting and
  capturing against local clones; commits queue in git history and drain on
  recovery.
</Callout>

## Choosing the transport

The link between the boxes carries exactly one thing: git over SSH, one deploy key per person, each key locked to its one repo by a forced command. Anything that can carry restricted SSH works, so choose by situation — including the situation where the agents and the brain run at different providers and you can't install a VPN on one side.

| Transport | Brain box exposure | Choose it when |
| --------- | ------------------ | -------------- |
| **Tailscale** (or any WireGuard-based mesh) | No public ports | You can install it on both hosts. Nice bonus: the same network carries your admin SSH and the dashboard tunnel. |
| **Provider private network** | No public ports | Both hosts sit at the same provider and region — the private LAN is free and there's nothing extra to install or trust. |
| **Public SSH + deploy keys** | One public port (22), keys only | The hosts can't share a private network — agents on a managed platform, a different provider, or anywhere you can't install a VPN. The forced-command restriction does the same work either way. |
| **GitHub / Gitea as the hub** | None — the brain box only dials out | You'd rather not operate SSH at all. One private repo per person, per-repo deploy keys; both boxes push and pull against the forge. Costs: company knowledge rests at the forge, and sync becomes polling-shaped. |

These mix freely within one deployment: containers that share a private network with the brain box can use it, while an agent running somewhere else uses public SSH or the forge.

<Callout type="info">
  The security properties don't depend on the transport. One key still opens
  exactly one repo, the master still runs no agents, and validation is still
  asynchronous and fail-closed. The transport only decides what the network
  between the boxes looks like.
</Callout>

## Sizing

| Headcount | Agents box RAM |
| --------- | -------------- |
| ~5        | 4 GB           |
| ~15       | 8 GB           |
| ~30       | 16 GB          |

The driver is one idle hermes gateway process per person (≈300–400 MB, near-zero idle CPU). Containers themselves add roughly nothing — they share the base image and have no per-container VM. If headcount outgrows one box, split the compose file across two; nothing about the architecture changes.

## Visual surfaces (optional, phased)

Chat stays the primary product, but each container can expose visual surfaces. Sessions live in the container's shared `state.db`, so a conversation started on WhatsApp continues in the browser or desktop app — same brain, different doors.

| Surface | What the employee gets | Setup | Risk surface |
| ------- | ---------------------- | ----- | ------------ |
| **Chat** (default) | Capture, ask, share — the full agent | Pairing code, once | Nothing new |
| **Brain dashboard** | Read-only view of their own knowledge: what's new, search, the graph, a capture box | None — a URL | Lowest: read-only by construction |
| **Hermes web UI** | Browser chat with streaming, tool activity, sessions (`hermes serve`) | None — a URL | Full agent control — gate it well |
| **Hermes Desktop** | Native app attached to their container: chat, files, voice | Laptop install + sign-in | Same as web UI |

Add the **brain dashboard first** — it answers "what does my assistant actually know?" without handing anyone a control panel. Hermes Desktop attaches to a remote backend natively: the container runs `hermes serve`, and the user signs in under **Settings → Gateway**.

<Callout type="warning">
  Exposing `hermes serve` means remote control of that agent. Credentials must be
  **per-person, per-container** — never shared — and the auth gate must map each
  identity to exactly one container. Pilot: keep visual surfaces reachable only
  over your private network (tailnet or equivalent) or an SSH tunnel. Wider
  rollout: one SSO reverse proxy (OIDC / Nous OAuth) with per-person routes.
  The brain dashboard has no auth of its own — off-loopback it must sit behind the
  authenticated proxy.
</Callout>

## The life of a note

The whole system, traced through one piece of knowledge — twice.

**Capturing something private (the everyday case):**

1. **9:02 am — Alice's phone.** She voice-memos her bot: *"the client agreed to the March timeline."*
2. **Seconds later — her container.** Her agent transcribes it, writes a note into her Inbox in its local clone, commits, and replies "Saved." Already durable — this works even if the brain box is down.
3. **Within ~5 minutes — both boxes.** Her container pushes; the next [`brain cycle`](/reference/cli#brain-cycle) validates and applies it to the master and recompiles her vault.
4. **Any time after.** "What did we agree with the client?" — answered from the local index, instantly. *Visible to Alice only, at every step.*

**Sharing with the team (the human-gated case):**

1. **2:14 pm.** Alice: *"share the March timeline with the team."*
2. **Seconds later.** Her agent drafts a [promotion](/concepts/promotions) — nothing is shared yet; a draft is still hers.
3. **Within ~5 minutes.** The next cycle sweeps the draft into the master's pending queue. *Now visible to Alice + the approver.* Her own `Shares.md` note (generated into her vault) now lists it as awaiting approval — and will show the decision, including a rejection reason, once made.
4. **When a human decides.** The approver reviews and approves — the gate no agent can open.
5. **Next cycle.** The note materializes into the Team space; every affected vault recompiles.
6. **Within ~5 minutes.** Bob's container pulls; his agent can now answer with it. Alice's agent tells her it went live. *Now visible to the whole team.*

<Callout type="info">
  **The two-sentence mental model for employees:** anything you tell your
  assistant is yours alone, instantly. If you ask it to share something, a person
  approves it first — you'll get a message when it's live. Private knowledge
  moves at machine speed (~one cycle); shared knowledge costs a few
  machine-minutes plus one human decision — and that latency is the feature.
</Callout>

## The runbook

Phases 1–3 happen once and take an afternoon. Phase 4 is ~10 minutes per new employee. Phase 5 is the steady state.

<Steps>
  <Step title="Provision — once">
    Two VPSes — same provider and region if you can, so the private LAN is an
    option. Ubuntu LTS, key-only SSH, unattended-upgrades. Then connect them
    with [whichever transport fits](#choosing-the-transport) and open no other
    inbound ports. (One documented exception exists: opting into webhook
    intake, below, adds a single TLS-fronted public port on the brain box.)

    ```bash
    # e.g. with Tailscale, on both boxes:
    curl -fsSL https://tailscale.com/install.sh | sh
    tailscale up   # names: brain-box, agents-box

    # or with a provider private network / public SSH: no install —
    # just make "brain-box" resolve on the agents box:
    echo "10.0.0.2  brain-box" >> /etc/hosts
    ```
  </Step>

  <Step title="Brain box — once">
    Install brainkit, scaffold the master, and let cron be the whole
    orchestration layer (see [Getting Started](/getting-started) for the
    `org.yaml` / `spaces.yaml` details):

    ```bash
    uv tool install git+https://github.com/joedanz/brainkit  # brainkit is not on PyPI
    brain init /srv/brain/master --company "Acme Co"
    brain compile --master /srv/brain/master --out /srv/brain/compiled
    brain doctor  --master /srv/brain/master --out /srv/brain/compiled
    adduser --disabled-password brain-sync   # per-person keys, one repo each
    ```

    ```bash
    # crontab
    */5 * * * * brain cycle --master /srv/brain/master \
      --out /srv/brain/compiled --index --json \
      >> /var/log/brain/cycle.jsonl || curl -fsS $ALERT_URL
    17 3 * * *  brain doctor --master /srv/brain/master \
      --out /srv/brain/compiled || curl -fsS $ALERT_URL
    23 4 * * *  git -C /srv/brain/master push --mirror offsite   # backup
    ```

    A nonzero `cycle` means a rejected write-back; a nonzero `doctor` means an
    integrity error — both ping the alert URL. Optionally set an
    [embedding provider](/reference/configuration#embeddings) so `--index`
    builds semantic (not just keyword) indexes.

    **Recommended — webhook intake.** A brain that only receives what people
    remember to send it starves; letting services push — every Fathom meeting
    transcript, a Zapier or Composio trigger on email and chat — is usually the
    single highest-value intake. `brain init` scaffolds a commented
    [`_meta/webhook.yaml.example`](/reference/configuration#webhookyaml) next to
    `org.yaml`; copy it to `webhook.yaml`, fill in your sources, and run
    [`brain webhook`](/reference/cli#brain-webhook) on this box as a supervised
    service with a TLS reverse proxy in front of its loopback port. Both halves
    ship ready to install in
    [`deploy/brain-box/`](https://github.com/joedanz/brainkit/tree/main/deploy/brain-box):
    `brain-webhook.service` (systemd, secrets from an `EnvironmentFile`) and
    `Caddyfile.webhook` (TLS, exposes only `/hook/*` and `/healthz`).

    Go in eyes open: outside providers can't reach your private network, so
    this is the one place the brain box gains a public HTTPS surface. The receiver
    authenticates every request before parsing a byte, every delivery lands
    through the same `brain ingest` primitive the cron loop already audits, and
    the failure drills below bound the worst case — but if your deployment
    genuinely has no push sources, skip it and keep zero inbound.
  </Step>

  <Step title="Agents box — once">
    Install Docker and build the `hermes-brain` image — it ships in the repo at
    [`deploy/agents-box/`](https://github.com/joedanz/brainkit/tree/main/deploy/agents-box),
    based on the official `nousresearch/hermes-agent` image with brainkit baked
    in. One compose stanza per person:

    ```yaml
    services:
      agent-alice:
        image: hermes-brain:latest
        restart: unless-stopped
        command: gateway run
        environment:
          - BRAIN_PERSON=alice
          - BRAIN_GIT_REMOTE=brain-sync@brain-box:/srv/brain/compiled/alice  # "brain-box" = the brain host on your chosen transport
          - TELEGRAM_BOT_TOKEN=${ALICE_BOT_TOKEN}
          - ANTHROPIC_API_KEY=${LLM_KEY}
          - BRAIN_EMBED_BASE_URL=${EMBED_URL}  # optional
        volumes:
          - alice-state:/opt/data   # hermes state: memory, sessions, pairing, deploy key
          - alice-vault:/vault      # her compiled vault clone
    ```

    Each container's first boot installs the `company-brain` hermes profile
    (SOUL.md, `terminal.cwd: /vault`, the brain MCP server and skills) and
    **generates its own deploy key** — the public half lands in `docker logs`,
    the private half never leaves the container. A supervised sync loop then
    clones `$BRAIN_GIT_REMOTE`, runs [`brain index`](/reference/cli#brain-index),
    and keeps syncing: pull → index → push every five minutes.
  </Step>

  <Step title="Onboard an employee — ~10 minutes">
    `deploy/agents-box/add-agent.sh <name>` prints the compose stanza and this
    exact checklist:

    1. Add them to `org.yaml`; the next cycle compiles their vault. Set
       `receive.denyCurrentBranch updateInstead` on the compiled repo so it
       accepts the agent's pushes.
    2. Telegram: `@BotFather → /newbot` → copy the token (WhatsApp: link a
       number via the hermes WhatsApp adapter).
    3. Agents box: paste the compose stanza, `docker compose up -d`.
    4. `docker logs agent-<name>` shows the container's deploy key — append one
       `authorized_keys` line on the brain box (the `brain-serve-repo` forced
       command locks the key to that one repo). The clone lands within 30s.
    5. They message their bot; approve the one-time pairing code.
    6. Send them one line: *"This is your assistant — tell it things, ask it
       things."* That's their entire onboarding.
  </Step>

  <Step title="Operate — minutes per week">
    - **Approvals:** `ssh brain-box brain promotions approve <id> --master /srv/brain/master --approver <you>`
      (`<you>` is your person id from `org.yaml`); or review in the dashboard
      over an SSH tunnel and approve there — pick your name from the dropdown.
    - **Watch two numbers:** the cycle and doctor exit codes (already wired to
      your alert URL).
    - **Backups, both halves:** the master mirror covers knowledge; agent
      memories, sessions, pairing, and deploy keys live on the **state volumes** —
      back them up nightly with [`deploy/agents-box/backup-agents.sh`](https://github.com/joedanz/brainkit/blob/main/deploy/agents-box/backup-agents.sh)
      (`hermes backup` inside each container, consistent even while the gateway
      runs) and ship the zips off-box encrypted — they hold keys and chat history.
      Vault volumes need no backup: they re-clone from the brain box.
    - **Offboarding:** remove from `org.yaml`, delete their `authorized_keys`
      line, remove their container.
  </Step>
</Steps>

## Failure drills

| Scenario | Outcome |
| -------- | ------- |
| Brain box down | Agents keep chatting and capturing on local clones; commits drain on recovery. Users notice nothing. |
| Agents box down | Chat goes quiet; zero data lost. `docker compose up -d` restores everything from volumes. |
| Agent volumes lost | Fresh container, `hermes import <zip> --force` with the person's latest backup, then `docker restart` (re-fixes key file modes) — memories, pairing, and deploy key all return; the vault re-clones itself. |
| Agents box compromised | Rotate deploy keys, rebuild containers from the image — the master's history is intact and trustworthy. |
| Brain box lost | Restore the offsite mirror, recompile, re-point remotes. |
| Webhook secret leaked (if webhook intake is enabled) | Worst case is note spam into the Inboxes that source routes to, capped at the source's rate limit (60/minute by default) — a webhook source can never read anything or write outside an Inbox, and past the cap deliveries get `429`s while the receiver logs a warning. Rotate the `secret_env` value and restart the receiver; the old secret is dead immediately. Every accepted delivery is its own audited `Brain Ingest` commit, so cleanup is `git revert`. |
| Webhook endpoint probed | Unauthenticated requests are refused before the payload is parsed (`401`/`404`), nothing is written, and oversized or compressed bodies are rejected outright. The receiver holds no read paths — there is no query to leak a note through. |
