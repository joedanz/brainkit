<Callout type="warning">
  **Agent profiles are not a filesystem sandbox.** A Hermes profile isolates
  config, memory, and sessions — but the agent still has the operating-system
  user's full file access. If two employees' agents share one uncontained host,
  each can read the other's vault. The profile boundary is not the tenant wall.
</Callout>

The tenant wall in this system is *physical*: it holds only where a personal agent can literally reach one person's files and no one else's. So a personal agent runs only in one of these two places:

<CardGroup cols={2}>
  <Card title="On the employee's own device" icon="laptop">
    Against their synced vault. This is the default and the simplest safe option
    — the OS account already scopes file access to that person.
  </Card>
  <Card title="Server-side, in a container" icon="box">
    A container (e.g. a Hermes Docker terminal backend) that mounts **only** that
    person's compiled vault. The mount is the boundary.
  </Card>
</CardGroup>

**Never** co-host multiple employees' agent profiles side-by-side on one uncontained host. That is the single configuration that turns a structural guarantee back into a promise an agent could break.

  Want the container branch end-to-end? The
  [two-box reference deployment](/guides/reference-deployment) is the worked
  example: chat-only users, one hermes container per person, the sync plane,

</Callout>

## Why the compiler doesn't save you here

The compiler guarantees that a person's *vault* contains only what they may see. That guarantee is about what gets written to disk. It says nothing about what a process running as your OS user can `open()` elsewhere on that disk. Keeping agents to one-vault-per-host (device or container mount) is what preserves the boundary at runtime.

## Server-side jobs are different

The **server assistant** agent is *supposed* to have full master access — it ingests transcripts, routes knowledge, and maintains dashboards across all spaces. Run it with master access, on cron and on Inbox drops. The deployment rule above is about *personal* agents, which must never see more than their one slice.

The mechanical half of that server-side loop is a single command on a schedule:

```bash
# refresh everyone: write back edits, sweep drafts, recompile (+ reindex)
*/10 * * * * brain cycle --master /srv/brain/master --out /srv/brain/compiled --index --json >> /var/log/brain-cycle.log 2>&1

# integrity check — nonzero exit on any error finding
0 * * * * brain doctor --master /srv/brain/master --out /srv/brain/compiled --json >> /var/log/brain-doctor.log 2>&1
```

[`brain cycle`](/reference/cli#brain-cycle) runs write-back → promotion sweep → recompile in the one order that's safe; a rejected write-back exits nonzero without halting the refresh. The `--index` flag also refreshes each vault's [search index](/concepts/retrieval) — worth it when agents run server-side (container mounts) rather than on employee devices; an index failure only warns, it never fails the cycle. [`brain doctor`](/reference/cli#brain-doctor) is the read-only health check — point it at both the master and the compiled root so it can catch a `_meta/` leak or a crashed-compile tombstone before anyone syncs.

<Callout type="info">
  A server that holds the master needs a backup story too — the master vault is
  the one thing nothing can rebuild. [Backups & Restore](/guides/backup-restore)
  covers what to protect, the encrypted-offsite scripts in `deploy/`, and the
  restore drill for each failure.
</Callout>

<Callout type="info">
  Server-side indexing needs an [embedding provider](/reference/configuration#embeddings)
  in the cron environment, and its cache lives at `_meta/cache/` so a chunk
  shared across people embeds once. If no provider is set, `--index` still
  builds keyword-only indexes. Note that configuring a provider means note text
  is sent to that endpoint — use a self-hosted or company-proxy endpoint if that
  matters.
</Callout>
