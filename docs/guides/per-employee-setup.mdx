Once [the master is compiled](/getting-started), each employee needs two things: sync access to *their* vault (and nothing else), and an agent pointed at it.

<Steps>
  <Step title="Grant per-person sync access">
    Give the employee access to `compiled/<person-id>/` **only** — a per-person
    deploy key or a per-person remote. Never hand out access to the master or to
    another person's slice.

    <Callout type="warning">
      This is a real boundary, not a convention. The master vault is the tenant
      wall; a person who can read the master can read everyone. Sync access is
      always to a compiled slice.
    </Callout>
  </Step>

  <Step title="Clone and open in Obsidian">
    On their machine:

    ```bash
    git clone <their-vault-remote> ~/brain
    ```

    Open `~/brain` as an Obsidian vault (the pilot uses the Obsidian Git plugin
    to sync). Everything is plain markdown, so any editor works too.
    Obsidian's built-in graph view works out of the box — see [See your brain as a graph](/guides/graph-view).
  </Step>

  <Step title="Point an agent at the vault">
    The compiled vault is self-describing — its root `AGENTS.md` and `CLAUDE.md`
    carry the full protocol.

    <Tabs>
      <Tab title="Claude Code">
        Nothing to install. Open `~/brain` in Claude Code; the vault's
        `CLAUDE.md` carries the routing and promotion protocol.
      </Tab>
      <Tab title="Hermes">
        ```bash
        hermes profile install github.com/<your-org>/company-brain --alias
        ```
        Set `terminal.cwd` to `~/brain` in the profile config before the first
        run. Credentials and sessions stay on the employee's machine. The
        profile ships with hermes's built-in memory **disabled** — a native
        memory tool would win every "remember this" over the vault, and facts
        stored there never sync. The vault is the agent's only durable memory.
      </Tab>
    </Tabs>
  </Step>

  <Step title="Build the search index and register it as a tool">
    ```bash
    brain index --vault ~/brain
    claude mcp add brain -- brain mcp --vault ~/brain
    ```

    `brain index` builds a local hybrid [search index](/concepts/retrieval) so
    the agent can query the vault instead of reading files by hand; the MCP
    server exposes it as `brain_search` / `brain_read` / `brain_links` /
    `brain_graph` / `brain_recent` / `brain_facts` tools. The index is
    keyword-only until an embedding provider is configured
    ([`BRAIN_EMBED_*`](/reference/configuration#embeddings)) — set one to add
    semantic search.

    <Callout type="info">
      `.brain/` is machine-local and gitignored, so it never syncs — each device
      builds its own index. Because it's built only from this compiled slice, it
      can never surface anything the person can't already read.
    </Callout>
  </Step>
</Steps>

## The daily flow

- **In:** transcripts and notes reach `People/<you>/Inbox/` — dropped in as a
  file, sent through an [email/chat/voice/upload channel](/guides/getting-things-in)
  backed by `brain ingest`, or pushed automatically by an outside service via
  [`brain webhook`](/guides/getting-things-in#webhooks--automatic-intake-from-other-services).
  The agent summarizes, extracts decisions and action
  items, and routes them per the vault protocol. Articles, links, PDFs, and
  screenshots get [distilled into the shared `Company/Intel/` wiki](/guides/getting-things-in#articles-and-links-the-intel-wiki),
  never archived whole.
- **Back up:** edits sync to the server, which runs a single scheduled command —
  [`brain cycle`](/reference/cli#brain-cycle) — that writes back every person's
  edits, provisions any self-service client requests, routes share requests
  (queuing shares, auto-applying revokes), sweeps agent drafts into the pending
  queue, and recompiles everyone, in that order. (A rejected edit reverts on the
  recompile and the command exits non-zero — as does an owner-mismatch client or
  share request — so cron alerts.)
- **Sharing:** the agent drafts a promotion; an admin approves it with
  `brain promotions approve <id> --master … --approver admin` — the approver is
  an admin-role identity from `org.yaml` (a dedicated operator account, not an
  employee); anything else is refused. See [Promotions](/concepts/promotions). A
  space's owner can also share or revoke a colleague's access to that space
  directly — a separate, owner-initiated flow from the admin-curated promotions
  gate — via [`brain shares`](/reference/cli#brain-shares); `share` requests
  still queue for approval by their decider — the recipient for a person-share,
  a team lead for a team-share, an admin for company-wide — while `revoke`
  requests apply immediately. See
  [Sharing a space](/concepts/spaces-and-permissions#sharing-a-space).
- **Search:** after each pull, the agent runs `brain index --vault ~/brain`
  (cheap — only changed files re-embed) and queries the vault through the
  `brain_search` MCP tool. See [Retrieval](/concepts/retrieval).

Read the [deployment rule](/guides/deployment) before running any personal agent
server-side — it is the one operational mistake that breaks the tenant wall.
