The `brain` command drives the whole lifecycle: scaffold a master, compile slices, apply edits back, manage promotions, run the whole cycle on a schedule, and check integrity.

```
brain {init,rename-entities,ingest,webhook,compile,writeback,promotions,shares,cycle,index,search,graph,facts,mcp,status,dashboard,doctor} …
```

Every subcommand exits `0` on success and `1` on a handled error (unknown person, rejected write-back, invalid promotion, or an integrity error), with a human-readable message on stderr. `main(argv)` is importable and returns the same code, so the CLI is testable without a subprocess.

## `brain init`

Scaffold a new company master vault.

```bash
brain init <dir> --company "Acme Co"
```

| Argument | Required | Meaning |
|---|---|---|
| `dir` | yes | Destination directory; must be empty or nonexistent. |
| `--company` | yes | Company name, woven into the seeded `Company/` notes. |
| `--entities` | no | Top-level folder name for the restricted third-party tree (default `Clients`). |
| `--entity` | no | Singular form for the entity type, used in guidance and folder names (default derived from `--entities` by stripping trailing "s"). |

Creates the space layout — including the seeded `Company/Intel/` reference wiki (`Destinations/`, `Providers/`, `Events/`, `Trends/`, mapped in `Intel/Home.md`) — `_meta/org.yaml` and `_meta/spaces.yaml`, the `_meta/promotions/{pending,approved,rejected}/` queue, the master-root `AGENTS.md` (assistant protocol), and an initial git commit.

## `brain rename-entities`

```bash
brain rename-entities --master <master> --entities Vendors [--entity vendor]
```

Renames the third-party tree of an existing vault: moves the folder (via
`git mv`), rewrites the affected `spaces.yaml` rules, moves each person's
request folder (`ClientRequests/` -> `VendorRequests/`, rewriting the
frontmatter key in pending requests), rewrites `space:`/`target-path:` values
in *pending* share and promotion queues (decided history is never touched),
writes `_meta/config.yaml`, and commits. Run it immediately after `brain
cycle`, then run a cycle again: slices recompile under the new name. Edits
still sitting in a stale slice under the old name are rejected fail-closed by
that next cycle and reported.

## `brain ingest`

Write one provenance-stamped note into a person's `Inbox/` in the master. The safe, server-side intake used by [email, chat, voice, and upload channels](/guides/getting-things-in) — and by [`brain webhook`](#brain-webhook) for services that push.

```bash
brain ingest --master <master> (--person <id> | --from <email>) [--file <path>] [--title <t>] [--source <channel>] [--date <YYYY-MM-DD>] [--json]
```

| Flag | Required | Meaning |
|---|---|---|
| `--master` | yes | Path to the master vault. |
| `--person` | one of | Target person by id. |
| `--from` | one of | Target person by their `org.yaml` `email` (unknown senders are refused). Mutually exclusive with `--person`. |
| `--file` | no | Read the note body from this file; default is stdin. |
| `--title` | no | Note title. Defaults to the `--file` name, else the first non-empty line, else `note`. |
| `--source` | no | Intake channel label (`email`/`chat`/`voice`/`upload`/`manual`). Default `manual`. |
| `--date` | no | Provenance date. Default today. |
| `--json` | no | Emit `{"person_id", "rel_path", "title", "source", "committed", "ok"}`. |

The note lands at `People/<id>/Inbox/<date>-<slug>.md` — the path is constructed (never derived from input), then re-checked with `can_write_path`, so intake can only ever write into that person's own Inbox. Provenance metadata is rejected if it contains newlines (no frontmatter injection). Each intake is a separate git commit under `Brain Ingest <ingest@brain.local>`, distinct from write-back. Exactly one of `--person`/`--from` is required; passing neither or both is a usage error (exit `2`). An unknown person/sender or an unwritable target exits `1`.

## `brain webhook`

Serve the authenticated webhook intake receiver — the always-listening counterpart to the [channel recipes](/guides/getting-things-in#webhooks--automatic-intake-from-other-services). External services (Fathom, Composio triggers, Zapier, Make, n8n) POST to `/hook/<source-id>`; each verified delivery becomes one `brain ingest` into the routed person's Inbox.

```bash
brain webhook --master <master> [--host <addr>] [--port <n>] [--max-body-mb <n>]
```

| Flag | Required | Meaning |
|---|---|---|
| `--master` | yes | Path to the master vault (sources come from `_meta/webhook.yaml`). |
| `--host` | no | Bind address. Default `127.0.0.1`; a non-loopback bind expects a TLS reverse proxy in front. |
| `--port` | no | Port. Default `8766`; `0` picks a free port. |
| `--max-body-mb` | no | Largest accepted request body in MB. Default `5` (transcripts are large). |

Sources are declared in `_meta/webhook.yaml` — server-only, never compiled into any vault. Each source sets a verify mode (`standard-webhooks` for HMAC-signed senders with replay protection, or `token` for a shared-secret header), names its secret's **environment variable** (secrets never live in the file), and routes deliveries either to a fixed `person:` or by `route: sender-email` lookup against `org.yaml` — unknown senders are refused. A source whose secret variable is unset refuses to start. Failed verification is `401`, an unknown source path is `404`, an unroutable or malformed payload is `400`, and duplicate deliveries (by `webhook-id`) are acknowledged without re-ingesting. Compressed request bodies are refused; the body-size cap is enforced before buffering.

## `brain compile`

Materialize per-person vaults from the master.

```bash
brain compile --master <master> --out <out> [--person <id>]
```

| Flag | Required | Meaning |
|---|---|---|
| `--master` | yes | Path to the master vault. |
| `--out` | yes | Output root; each person's slice lands in `<out>/<id>/`. |
| `--person` | no | Compile only this person instead of everyone. |

<Callout type="info">
  Re-run on every master change. The compiler [fails closed](/concepts/the-compiler#fail-closed):
  if a compile errors, the previous slice stands.
</Callout>

## `brain writeback`

Validate a person's edits and apply them to the master.

```bash
brain writeback --master <master> --vault <vault> --person <id>
```

| Flag | Required | Meaning |
|---|---|---|
| `--master` | yes | Path to the master vault. |
| `--vault` | yes | The person's edited (synced) vault. |
| `--person` | yes | Whose edits these are — determines the write scope. |

Every changed path is checked against the person's write permissions server-side. If **any** path is out of scope, the whole change set is rejected: nothing is applied, the violations print to stderr, and the command exits `1`. This is deliberate — there is no partial write that could leave the master half-updated.

## `brain promotions`

Manage the [promotion queue](/concepts/promotions).

```bash
brain promotions {list,show,sweep,approve,reject} [id] --master <master> [--approver <who>] [--reason <text>]
```

| Action | Uses | Meaning |
|---|---|---|
| `list` | `--master` | Print pending promotions (`id`, from, target). |
| `show <id>` | `--master` | Print a pending promotion's metadata (`mode`, target, source) and, for `mode: patch`, a live unified diff of the proposed body against the current target — computed at review time, not from the queued draft. |
| `sweep` | `--master` | Pull agent drafts from `People/*/Promotions/` into the pending queue. For `mode: patch` drafts, stamps `base-hash` (a sha256 of the target file, taken at sweep time) into the queued promotion. |
| `approve <id>` | `--master --approver` | Publish the promotion to its target space with provenance frontmatter. |
| `reject <id>` | `--master --reason` | Move it to `rejected/` with the reason. |

`approve` requires `--approver`, and it must be a person id from `_meta/org.yaml` — a blank or unknown name is refused, so the note's `approved-by` line always resolves to a real person. It also re-validates the target path at publish time, so a tampered pending file cannot write outside the master root or into a private space.

Every promotion carries a `mode`, chosen when the agent drafts it:

| Mode | Target must | On approval |
|---|---|---|
| `create` (default) | not already exist | writes the whole note as a new file with provenance frontmatter |
| `append` | already exist | adds the body under a `---` divider, with an italic attribution line naming who promoted and approved it |
| `patch` | already exist, unchanged since sweep | replaces the file wholesale with the promotion's body (the complete revised page) |

`create` and `append` fail closed on a target that doesn't meet their requirement — an approval can never silently create where it should update, or overwrite where it should add. `patch` fails closed if the target's sha256 no longer matches the `base-hash` stamped at sweep time: a hand edit, a bad merge, or another approval could have changed the page after this draft was queued, and re-checking at approval time (not draft time) is what catches it. Use `brain promotions show <id>` — or the dashboard's Promotions card — to review the live diff before approving a patch.

Every queue decision is its own git commit, scoped to just the files it touched: `sweep` and `reject` commit as `Brain Promotions`, and `approve` commits under the approver's identity — so `git log` answers "who published this, and when" the same way it does for ingest and write-back.

## `brain shares`

Manage [space shares](/concepts/spaces-and-permissions#sharing-a-space) — an owner granting or revoking a colleague or team's access to a space they own.

```bash
brain shares {list,approve,reject,revoke} [id] --master <master> [--approver <who>] [--reason <text>] [--space <path>] [--subject <grant>]
```

| Action | Uses | Meaning |
|---|---|---|
| `list` | `--master` | Print pending share requests (`id`, from, space, share-with, access). |
| `approve <id>` | `--master --approver` | Amend `spaces.yaml` per the pending request; archive it to `approved/`. |
| `reject <id>` | `--master --reason` | Move it to `rejected/` with the reason. |
| `revoke` | `--master --space --subject` | Remove `--subject` (e.g. `person:yuki`) from `--space`'s rule directly. |

`list`, `approve`, and `reject` act on a **pending share id**, the same shape as `brain promotions`. `revoke` is different: there's no id, because it isn't a queued decision — it's a live admin action that takes the space and the subject to remove directly, and it doubles as the veto lever for a self-service-created client space (see [Self-service client spaces](/concepts/spaces-and-permissions#self-service-client-spaces)) or any other grant an admin wants to pull immediately, with no request in the queue.

`approve` requires `--approver`, a person id from `_meta/org.yaml` — a blank or unknown name is refused, the same rule as `brain promotions approve`. Both `approve` and `revoke` refuse to touch `role:admin` — it can never be removed from a rule. Every decision is its own git commit: `approve` under the approver's identity, `reject`/`revoke` as `Brain Shares`.

This command manages the *queue and direct revokes*; it doesn't create requests. An owner asks for a share or revoke by writing into their own `People/<id>/ShareRequests/`, and [`brain cycle`](#brain-cycle) routes it — `share` requests land in the pending queue this command works, `revoke` requests from that folder auto-apply during the cycle instead of ever reaching a queue. See [Sharing a space](/concepts/spaces-and-permissions#sharing-a-space) for the full request shape and the ownership rules `brain cycle` enforces on it.

## `brain cycle`

Run the whole server-side loop once: write back every person's edits, provision any self-service [client requests](/concepts/spaces-and-permissions#self-service-client-spaces), route pending/auto-applied [share requests](/concepts/spaces-and-permissions#sharing-a-space), sweep agent drafts into the promotions pending queue, then recompile all vaults.

```bash
brain cycle --master <master> --out <out> [--json] [--index]
```

| Flag | Required | Meaning |
|---|---|---|
| `--master` | yes | Path to the master vault. |
| `--out` | yes | Compiled-vault root — the same `--out` you pass to `compile`. |
| `--json` | no | Emit a single JSON summary object on stdout instead of human lines. |
| `--index` | no | After recompiling, refresh each vault's search index (see [`brain index`](#brain-index)). Index failures warn but never fail the cycle. |

### The `--json` summary

| Field | Meaning |
|---|---|
| `writebacks` | Per-person write-back results (`person_id`, `status`, `applied` count, `violations`). |
| `swept` | Promotion drafts pulled into the pending queue this cycle. |
| `compiled` | Vaults recompiled this cycle. |
| `pending` | Promotions still awaiting approval after this cycle. |
| `clients_created` | Self-service client spaces provisioned this cycle. |
| `clients_rejected` | Client requests refused this cycle — usually a routine name collision. |
| `clients_tampering` | Of those rejections, how many were an owner-mismatch — a tamper signal, not a routine outcome. |
| `shares_queued` | Share requests routed into `_meta/shares/pending/` for approval. |
| `shares_revoked` | Revoke requests auto-applied this cycle. |
| `shares_tampering` | Share/revoke requests from someone who doesn't own the space at cycle time — a tamper signal. |
| `indexed` | Vaults reindexed (only meaningful with `--index`). |
| `index_warnings` | Per-vault indexing warnings (only meaningful with `--index`). |
| `ok` | `false` if any write-back was rejected or a tampering signal fired — see below. |

The ordering is load-bearing: write-back lands each person's edits (including freshly synced promotion, client, and share requests) in the master *first*. `materialize_clients` provisions any self-service client spaces next, so a freshly granted space is visible this same cycle. `sweep_shares` then routes share/revoke requests — queuing shares, auto-applying revokes. Both of those steps can append to `spaces.yaml`, which is reloaded before the promotions sweep and the compile so they see the fresh rules. The promotions sweep runs after that, and the recompile runs *last* so every vault reflects the fully updated master. A rejected write-back never halts the cycle — the rest of the pipeline still runs, and the fresh compile reverts the rejected edits server-side (fail closed).

`ok` is `false` when any write-back is rejected, or when a client or share request is flagged as **tampering** — an owner-mismatch client request, or a share/revoke request from someone who turns out not to own the space when the cycle runs. Routine outcomes don't flip it: a client request that loses a name collision (`clients_rejected`) or a share request rejected for an ordinary reason (unknown recipient, already shared, nothing to revoke) is a normal user outcome, surfaced via the requester's own inbox note — not a security event. `--index` adds `indexed` and `index_warnings` to the summary; because retrieval is a convenience layer and compile is the security layer, a failed index only warns and never flips `ok`.

<Callout type="info">
  This is the one command to run on a schedule — a cron entry or a git
  post-receive hook. It replaces running `writeback`, `promotions sweep`, and
  `compile` by hand. A person whose vault has never been compiled is `skipped`
  (not an error); the recompile creates it. A person whose manifest is corrupt
  is likewise `skipped` (with the reason in `violations`) rather than aborting
  everyone else's refresh — the recompile rewrites a clean manifest, so the
  next cycle recovers on its own.
</Callout>

```bash
# every 10 minutes
*/10 * * * * brain cycle --master /srv/brain/master --out /srv/brain/compiled --json >> /var/log/brain-cycle.log 2>&1
```

## `brain index`

Build or refresh the hybrid search index for one compiled vault. See [Retrieval](/concepts/retrieval) for how the index inherits the compiler's boundary.

```bash
brain index --vault <vault> [--full] [--json]
```

| Flag | Required | Meaning |
|---|---|---|
| `--vault` | yes | A compiled vault (must contain `.brain-manifest.json`). |
| `--full` | no | Rebuild from scratch instead of diffing the manifest. |
| `--json` | no | Emit a JSON report (`files_indexed`, `chunks_embedded`, `chunks_from_cache`, `mode`, …). |

Incremental by default: only files whose manifest sha changed are re-chunked, and only cache-miss chunks are embedded. Embeddings come from the configured provider (see [Configuration](/reference/configuration)); with none set, indexing is **keyword-only** — still fully searchable, just no semantic leg. A model change forces a full rebuild.

## `brain search`

Query a compiled vault's index. Fuses keyword (FTS5) and semantic (vector) ranking with reciprocal-rank fusion.

```bash
brain search "<query>" --vault <vault> [--k 8] [--keyword-only] [--center <rel-path>] [--json]
```

| Flag | Required | Meaning |
|---|---|---|
| `query` | yes | The search query (positional). |
| `--vault` | yes | The compiled vault to search. |
| `--k` | no | Max results (default 8). |
| `--keyword-only` | no | Skip the vector leg even if a provider is configured. |
| `--center` | no | A note's relative path; results near it in the wikilink graph rank higher (see [Retrieval](/concepts/retrieval)). |
| `--json` | no | Emit `{"query", "mode", "hits": [...], "warnings"}`. |

Each hit carries the file path, heading path, a highlighted snippet, a fused score, and which legs matched. A missing index exits `1` with the exact `brain index` command to build it.

## `brain graph`

Walk a note's typed relations in a compiled vault. Shows the neighborhood up
to `--depth` hops, each edge labeled with its relation (`up`, `down`, `same`,
`prev`, `next`) and why it exists (declared in frontmatter, an automatic
inverse, or mined from folders, dates, or shared entity types).

```bash
brain graph "Clients/Acme/Kickoff.md" --vault out/joe --rel up --depth 3
```

| Flag | Required | Meaning |
|---|---|---|
| `note` | yes | Rel path, or a filename stem (positional). |
| `--vault` | yes | Compiled vault root. |
| `--rel` | no | Relation to follow, repeatable (default: all five). |
| `--direction` | no | `out` (declared on this note), `in` (declared elsewhere, pointing here), or `both` (default). |
| `--depth` | no | Hops to walk. Default `1`, max `5`. |

## `brain facts`

Query time-stamped fact lines — current, as-of a date, or believed-on a date — bullets carrying a `[from:: date]` (and optionally `[until:: date]`, `[source:: …]`) — from a compiled vault. See [Facts and time](/concepts/retrieval#facts-and-time) for the grammar and the current/as-of/believed-on distinction.

```bash
brain facts --vault <vault> [--entity <name>] [--type <type>] [--as-of <date>] [--believed-on <date>] [--ended] [--json]
```

| Flag | Required | Meaning |
|---|---|---|
| `--vault` | yes | The compiled vault to query. |
| `--entity` | no | Filter to one entity, matched by rel path, title, or alias. |
| `--type` | no | Filter to one entity type (e.g. `client`). |
| `--as-of` | no | Facts true on this date, using what the vault knows now. Default: today. |
| `--believed-on` | no | Answer from the vault's git history as of this date — what it would have said then, replayed at that commit. Mutually exclusive in effect with `--as-of` (this one wins if both are set). |
| `--ended` | no | Include facts that have since been closed with `[until::]`. |
| `--json` | no | Emit `{"facts": [...], "warnings": [...]}`. |

`--as-of` reads the current index (fast, reflects today's knowledge of the past); `--believed-on` re-parses the vault's files straight from git history at that date (no index needed, answers what was believed then). A vault with no index exits `1` for the default and `--as-of` forms; `--believed-on` needs only git.

## `brain mcp`

Run a stdio [MCP](https://modelcontextprotocol.io) server over a vault so any MCP client can search it as a tool.

```bash
brain mcp --vault <vault>
```

Register it once with a client — for Claude Code:

```bash
claude mcp add brain -- brain mcp --vault ~/brain
```

It exposes six read-only tools: `brain_search` (hybrid search, with an optional `center` note for graph-aware ranking), `brain_read` (read one file by relative path, refusing symlinks, `..` escapes, and anything outside a readable space), `brain_links` (a note's backlinks and outgoing wikilinks), `brain_graph` (a note's typed relations, each edge labeled with why it exists), `brain_recent` (recently changed notes, from the vault's git history), and `brain_facts` (time-stamped fact lines, current, as-of, or believed-on a date). It runs on the employee's device against their own vault clone, so it inherits the compiler's boundary. No HTTP, no OAuth — stdio only.

## `brain doctor`

Check a master — and optionally its compiled vaults — for integrity problems that otherwise fail silently. Read-only: `doctor` never mutates anything.

```bash
brain doctor --master <master> [--out <out>] [--json]
```

| Flag | Required | Meaning |
|---|---|---|
| `--master` | yes | Path to the master vault. |
| `--out` | no | Compiled-vault root. When given, adds per-vault checks (tombstones, `_meta/` leakage, manifest drift). |
| `--json` | no | Emit `{"ok": bool, "findings": [...]}` on stdout instead of human lines. |

Findings carry one of three severities:

| Severity | Meaning | Examples |
|---|---|---|
| `error` | An invariant is broken. Exits `1`. | Malformed `org.yaml`/`spaces.yaml`, an unknown `person:` in a rule, a symlink in the master, a crashed-compile tombstone, an unreadable manifest, or `_meta/` leaked into a compiled vault. |
| `warn` | Probably a mistake, but nothing leaks. | An empty team/role, a rule path that matches nothing, a space no rule covers, a space **readable by no one** (its rule resolves to zero readers — e.g. a folder named `Teams/Sales` when the team id is `sales`, or a departed person's `People/` folder — so its notes compile into no vault), a `.md` sitting loose under `Teams/`/`People/`/`Clients/` (in no space, so compiled into no vault), a **cross-space reference** (a wikilink) or **plain-text mention** (a restricted space's name in shared prose) whose name is visible to readers who can't see its home space, a promotion draft the sweep will silently skip (a target path it can't validate, an unrecognized `mode`, or a `mode: patch` draft whose target is a symlink or doesn't exist), or a `Company/Intel/` page that's fallen out of the wiki's conventions — its newest `as of`/`captured` citation over 12 months old, no dated citation at all (`Home.md` exempt), or a lingering `<Name> — updates YYYY-MM.md` addendum never folded back in, or a note with no links, typed relations, or facts connecting it to anything (`unlinked-notes`) — invisible to the graph leg of retrieval until someone links it (notes under an `Inbox/` folder are exempt). |
| `info` | Normal state worth seeing. | Files awaiting write-back (vault drift), a count of promotions pending approval, one finding per self-service-created client space (`clients` — for admin review: rename, merge, or revoke), one finding per share request awaiting approval (`shares`). |

Exit is `1` if any `error` finding exists, else `0` — so `doctor` slots into CI or a pre-cron gate. Run it after setup and on a schedule.

<Callout type="info">
  Vault drift — a compiled file whose contents differ from the manifest
  baseline — is the *normal* "edits awaiting write-back" state, so it is
  reported as `info`, never an error.
</Callout>

## `brain status`

Counts, freshness, and health at a glance — in the terminal. Read-only: index reads go through a read-only SQLite connection, so `status` can never create or migrate an index as a side effect. With an index present, the output includes facts and entities counts.

```bash
brain status --vault <vault> [--json]                  # user lens
brain status --master <master> [--out <out>] [--json]  # admin lens
```

| Flag | Required | Meaning |
|---|---|---|
| `--vault` | one of the two | A compiled per-person vault. Shows notes/chunks by space, index freshness, files pending reindex, inbox/open-action counts, most-linked notes, and embedding coverage. |
| `--master` | one of the two | The master vault. Shows people, spaces, the promotion queue, a read/write permission matrix, and all `doctor` findings. |
| `--out` | no | Compiled-vault root (admin lens only). Adds a per-person table: vault size, note count, index age, and edits awaiting write-back. |
| `--json` | no | Emit the full stats structure on stdout — the automation surface. |

`status` is informational and exits `0` even with warnings; `doctor` remains the failing check for CI.

<Callout type="info">
  A vault without an index still reports space and note counts from the
  manifest, plus a hint to run `brain index`. Nothing here requires the
  embedding provider to be configured.
</Callout>

## `brain dashboard`

Serve a **live dashboard** that updates in real time as the brain changes, or — with `--html` — write the original **self-contained HTML snapshot** (one file, inline styles and scripts, no network requests) for emailing or archiving. The user dashboard includes a Facts tab (filter by entity, type, or date), and the 2D graph rings typed entity pages by type.

```bash
# live server (default): starts on http://127.0.0.1:8765 and opens a browser
brain dashboard --vault <vault>
brain dashboard --master <master> --out <out>

# static snapshot (original behavior): write one HTML file and exit
brain dashboard --vault <vault> --html dash.html --open
```

| Flag | Required | Meaning |
|---|---|---|
| `--vault` / `--master` | one of the two | Same two lenses as `status`. |
| `--out` | no | Compiled-vault root (admin lens; enables per-person graph/query). |
| `--html PATH` | no | Write a static self-contained HTML file to `PATH` and exit, instead of serving. |
| `--host` | no | Live server bind address (default `127.0.0.1`). |
| `--port` | no | Live server port (default `8765`; `0` picks a free port). |
| `--no-open` | no | Do not open a browser when serving. |
| `--open` | no | Open the file in a browser (static `--html` mode only). |

<Callout type="warning">
  **Behavior change:** a bare `brain dashboard --vault X` now *serves* the live
  dashboard rather than writing `brain-dashboard.html`. Scripts that relied on
  the old default file must pass `--html brain-dashboard.html` explicitly.
</Callout>

The live dashboard is organized into tabs. The **user lens** (`--vault`) shows:

- **Overview** — a lead figure and, only when something is actually pending, a "needs attention" list (inbox, open actions, notes awaiting reindex) over notes-by-space, most-linked notes, and recent activity.
- **Inbox** and **Actions** — the "what needs me" worklists: notes waiting in your Inbox (newest first) and every open `- [ ]` action across your Actions notes, each row opening the source note.
- **Graph** — an interactive D3 force graph of the vault's wikilinks (zoom/pan/drag, click a note to see what links in and out, a legend to toggle spaces, a highlight box, and a **3D view** toggle powered by three.js).
- **Query** — hybrid keyword+semantic search with structured filter chips (space, unresolved links, pending reindex, modified-after, path substring, keyword-only) and a **rendered** note reader: Markdown with clickable `[[wikilinks]]`, the note's **backlinks** (linked-from / links-to / unresolved), and a raw-text toggle. Keyboard: `/` focuses search, arrows move the highlight, Enter opens, Esc closes.

A **Capture** button (or `c`) is always in the header: jot a note straight into your Inbox. The composer names the exact destination (`People/<you>/Inbox`) before you save — under the user lens it writes into your compiled slice and write-back carries it to master on the next `brain cycle`; there is no way to write outside your own Inbox. From an open note you can also **Propose to share**, drafting a promotion an admin approves. The header carries a light/dark **theme toggle** (system by default, your choice persisted).

The **admin lens** (`--master`) adds tabs for **People** (per-person vault size, notes, index age, write-back drift), **Permissions** (a spaces × people read/write matrix), **Promotions**, and **Doctor** (integrity findings with severity badges). The **Promotions** tab is a live review surface: each pending item shows its destination, a warning naming exactly who will be able to read it, and its rendered body, with **Approve** / **Reject** (reason required) actions and a **Sweep drafts** button — the whole human gate the promotion queue exists for. Before approving you pick who you are from a dropdown of the org's people (your last pick is remembered), so `approved-by` is always a real person id, never a typo. A count badge on the tab surfaces new arrivals from any tab. The Graph and Query tabs gain a person selector that reads the chosen person's compiled vault under `--out`; Capture can file into any person's Inbox via the hardened `brain ingest` primitive.

Pending [space shares](/concepts/spaces-and-permissions#sharing-a-space) are collected into the master stats alongside promotions: `/api/stats` and `brain status --master --json` both carry a `shares_pending` array (`id`, `person_id`, `space`, `share_with`, `access`, `created`), and `POST /api/shares/{id}/{action}` (`approve`/`reject`) is the same primitive [`brain shares`](#brain-shares) uses — the static `--html` snapshot's admin view renders them in a **Share requests** section mirroring the Promotions table. The live tabbed dashboard doesn't yet give shares their own tab or Overview tile the way it does for promotions, and `brain status --master`'s human-readable output carries no shares line either (only its `--json` form does) — use `brain shares list` to see the queue, or `brain doctor` (one `info` finding per pending share) for a human-readable count, until the dashboard catches up.

As `brain cycle`/`brain index` run (or you commit to a vault), the page updates within a couple of seconds over a WebSocket — new notes spring into the graph and pulse, tiles retick, the connection dot stays green, and a disconnect shows a "data may be stale" notice until it self-heals.

<Callout type="info">
  Privacy stays structural: the user dashboard reads only the compiled vault, so
  it can only ever visualize what that person is allowed to see. Writes go
  exclusively through the same safe primitives the rest of the system uses —
  capture constructs a `People/<you>/Inbox/…` path (never caller input) and
  promotions still pass the human gate — so the dashboard can add to the brain
  without widening the trust boundary. The server binds `127.0.0.1`, rejects
  cross-origin `Host`/`Origin` headers, and requires a same-origin JSON `POST`
  for every write (CSRF defense); there is no authentication, so a non-loopback
  `--host` (which prints a warning) exposes the vault to anyone who can reach the
  port. `org.yaml`/`spaces.yaml` remain read-only — edit them in git.
</Callout>

## Exit codes

| Code | Meaning |
|---|---|
| `0` | Success. |
| `1` | Handled error — unknown person or sender, rejected write-back, invalid/missing promotion, an unwritable ingest target, or (for `doctor`) an `error`-severity finding. The message is on stderr. |
| `2` | Usage error from argument parsing (unknown subcommand, missing required flag). |
