Two files under `_meta/` describe the whole company: who exists (`org.yaml`) and who sees what (`spaces.yaml`) — plus an optional third, `webhook.yaml`, when you run the [webhook intake receiver](/guides/getting-things-in#webhooks--automatic-intake-from-other-services). All are plain YAML, validated on load — a malformed file is a clear error, never a silent misconfiguration. Nothing under `_meta/` is ever compiled into a person's vault.

## `org.yaml`

```yaml
people:
  alice: {name: Alice Nguyen, roles: [admin], teams: [sales], email: alice@acme.com}
  bob:   {name: Bob Rivera, teams: [ops], email: bob@acme.com}
```

| Field | Type | Required | Meaning |
|---|---|---|---|
| `people` | map of `id → person` | yes | The key is the person's id (used in vault paths and grants). |
| `name` | string | yes | Display name. |
| `roles` | list of strings | no | Roles this person holds; resolved against `role:` grants. Defaults to none. |
| `teams` | list of strings | no | Teams this person belongs to; resolved against `team:` grants. Defaults to none. |
| `email` | string | no | The person's address, used by [`brain ingest --from`](/guides/getting-things-in#email-in) and [`brain webhook`](/reference/cli#brain-webhook) sender-email routing to resolve intake to them. Must be unique across people; a duplicate is a load error. |

Field **types** are validated: a scalar where a list is expected (`roles: admin` instead of `roles: [admin]`) is rejected rather than silently coerced.

## `spaces.yaml`

```yaml
spaces:
  - {path: Company,     read: [everyone],        write: ["role:admin"]}
  - {path: "Teams/*",   read: ["team:{name}"],   write: ["team:{name}"]}
  - {path: "People/*",  read: ["person:{name}"], write: ["person:{name}"]}
  - {path: "Clients/*", read: [everyone],        write: ["role:admin"]}
```

| Field | Type | Meaning |
|---|---|---|
| `path` | string | The space path. A trailing `/*` makes it a wildcard over one segment. |
| `read` | list of grants | Who may read notes in this space. |
| `write` | list of grants | Who may write them. |

### Grants

| Grant | Matches |
|---|---|
| `everyone` | Every person in `org.yaml`. |
| `role:<name>` | Anyone whose `roles` include `<name>`. |
| `team:<name>` | Anyone whose `teams` include `<name>`. |
| `person:<id>` | Exactly that person. |

### The `{name}` binding

In a wildcard rule, `{name}` binds to the matched path segment. `Teams/*` with `read: ["team:{name}"]` means "the `Teams/sales` space is readable by the `sales` team" — one rule covers every team without enumerating them.

### Exact beats wildcard

When a space matches both an exact rule and a wildcard, the exact rule wins. Define `Teams/*` for the general case and add a specific `Teams/leadership` rule to carve one exception — the wildcard still governs every other team.

<Callout type="warning">
  A path that matches **no** rule resolves to no space. The compiler excludes it
  from every slice and write-back rejects edits to it. Unmatched is closed, not
  open — there is no implicit "readable by default."
</Callout>

## Changing the org

Adding a person, a team, or a space is an edit to these two files plus a `brain compile`. There is no migration step: the compiler reads the current `org.yaml` / `spaces.yaml` on every run, so the next compile reflects the change for everyone.

## `webhook.yaml`

Declares the sources [`brain webhook`](/reference/cli#brain-webhook) serves — opt-in to activate, but recommended for any deployment with push sources (meeting recorders, automation platforms). `brain init` scaffolds a commented `_meta/webhook.yaml.example`; copy it to `webhook.yaml` to enable. Each source is one URL path (`/hook/<id>`), one verification mode, and one routing rule:

```yaml
sources:
  - id: fathom-alice                # POST /hook/fathom-alice
    person: alice                   # fixed routing
    verify: standard-webhooks       # HMAC-signed senders (Fathom, Composio, …)
    secret_env: WEBHOOK_FATHOM_ALICE_SECRET
    source: fathom                  # provenance label; defaults to the id
    body_field: transcript          # payload field for the note body (dotted paths ok)
    title_field: title              # payload field for the title (optional)

  - id: zapier-intake
    route: sender-email             # resolve the person from the payload
    email_field: email              # …via this field, against org.yaml emails
    verify: token                   # shared-secret header (Zapier, Make)
    secret_env: WEBHOOK_ZAPIER_TOKEN
```

| Field | Required | Meaning |
|---|---|---|
| `id` | yes | URL path segment; lowercase letters, digits, hyphens. Must be unique. |
| `verify` | yes | `standard-webhooks` (HMAC-SHA256 signatures with replay protection) or `token` (`Authorization: Bearer` / `X-Brain-Token` header). |
| `secret_env` | yes | Name of the **environment variable** holding the secret. The secret itself never lives in this file; an unset variable refuses startup. |
| `person` | one of | Route every delivery to this person's Inbox. |
| `route: sender-email` | one of | Resolve the person from the payload's `email_field` against `org.yaml` emails — unknown senders are refused. Exactly one of `person`/`route` per source. |
| `email_field` | no | Payload field for sender-email routing. Default `email`. |
| `source` | no | Provenance label stamped on each note. Default: the source id. |
| `body_field` / `title_field` | no | Payload fields for the note body and title (dotted paths reach nested JSON). Defaults `body` / `title`. |

Validation fails closed: any invalid entry rejects the whole file, and `brain doctor` reports config errors and unset secret variables.

## Embeddings

Retrieval's semantic leg needs an embedding provider. Without one, `brain index` and `brain search` still work in **keyword-only** mode — configure a provider to add the vector leg. Any OpenAI-compatible `/embeddings` endpoint works (OpenAI, Together, a company proxy, a local `llama-server`).

Configure via environment variables:

| Variable | Default | Meaning |
|---|---|---|
| `BRAIN_EMBED_BASE_URL` | — | Endpoint base, e.g. `https://api.openai.com/v1`. **Set this to enable vectors.** |
| `BRAIN_EMBED_API_KEY` | — | Bearer token for the endpoint. |
| `BRAIN_EMBED_MODEL` | `text-embedding-3-small` | Embedding model id. |
| `BRAIN_EMBED_DIM` | `512` | Output dimension (sent as `dimensions`). |
| `BRAIN_EMBED_CACHE` | `~/.cache/brain/embeddings.db` | Embedding cache location. |

Or a config file at `~/.config/brain/config.yaml` (override the path with `BRAIN_CONFIG`):

```yaml
embeddings:
  base_url: https://api.openai.com/v1
  model: text-embedding-3-small
  dim: 512
  api_key_env: OPENAI_API_KEY   # the key is read from this env var, never stored here
```

<Callout type="warning">
  Configuring a provider means each note's text is sent to that endpoint to be
  embedded. On the server, `brain cycle --index` caches embeddings under
  `_meta/cache/` so a chunk shared across people is embedded once. A model
  change forces a full re-embed of every chunk.
</Callout>
