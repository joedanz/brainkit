Hermes Phase-1 Technical Build Plan — Luxury Travel Atelier

The hands-on Hermes deployment spec: the four-plane architecture, the dossier schema, the shadow-mode acceptance gate, per-quick-win build steps, and a 60-day sequence.

01ARCHITECTURE OVERVIEW

Four planes, structural boundaries, trust as the product

This is a host/network of 1099 independent-contractor advisors, the dominant luxury model. The architecture defaults everything to advisor-owned and makes the agency earn each act of sharing. Boundaries are enforced structurally at the ZeroID/MCP token layer (default-deny, time-boxed, row-level scope) — never by agent self-policing, because measured gate compliance is only ~112/129 and context compression drops constraints around turn 30. Self-hosted Hermes on a Mac mini or $10-20 VPS, with the gateway daemon installed as launchd/systemd.

Tier-0 — Agency Brain hermes primitive

Institutional data only, no voice. Runs the default Hermes instance (~/.hermes/) holding the advisor-owned dossier per-tenant BYOK-encrypted that it cannot read in plaintext — the host SELECTs return ciphertext. Owns state.db (FTS5 session store) and the shared supplier/duty-of-care logic. Never holds a SOUL.md persona.

Tier-1 — Chief-of-Staff Orchestrator cron + gateway

Routes work, mints scoped ZeroID tokens (one advisor_id, one TTL, one toolset), enforces the human-approval gate, and runs institutional cron jobs. The only plane allowed to fan work out to the Specialist Pool. Cron sessions cannot create more cron jobs (built-in safeguard).

Tier-2 — Personal Advisor Agents profiles

One per advisor, each its own ~/.hermes/profiles/<advisor>/ with its own config.yaml, .env, SOUL.md voice, memory, sessions, and state.db. Sees only assigned clients via the row-level token. House-voice and fact-promotion are strictly opt-in; non-response = no action. The advisor cannot have scope auto-widened.

Principal-Owner Agent + Specialist Pool

The Principal-Owner Personal Agent is walled from the admin hat, dual-controlled break-glass, with an externally-anchored, principal-uneditable audit. The Specialist Sub-agent Pool does delegated work with ephemeral PII and de-identified write-backs only.

Data flow (read)
Advisor message → Tier-2 Personal Agent → requests dossier read → Chief-of-Staff mints a ZeroID token scoped to advisor_id = self → SQLite MCP returns only that advisor's rows → agent drafts in house voice → advisor presses send.
Data flow (write)
Any mutation that moves money or makes a promise is staged, not committed: written to a draft surface (Gmail draft / held booking) and recorded in audit_log with before/after JSON. A human commits.
Shared-write path
Specialist Pool → deterministic PII redaction + k-anonymity floor (k≥5) → only then a shared/benchmark write. The host roll-up of individual books is excluded.
Kill switch
Token revocation is duty-of-care continuity, not control: a departing advisor triggers hot-handoff coverage before any token is revoked, so no trip lapses mid-flight.
The IC value proposition is baked in: "your clients are yours." Dossier + personal pipeline default advisor-owned, segregated, per-tenant encrypted, host-invisible, with one-click machine-readable export shipped as a product feature. Agency-wide scale lives only where it does not expropriate.

02THE LIVING CLIENT DOSSIER

Institutional memory in a structured DB, not flat markdown

Hermes' own docs are explicit: facts that must not hallucinate belong in a structured database, not the ~2.2k-char MEMORY.md snapshot that freezes at session start. The dossier is a separate SQLite DB the agent reaches via the SQLite MCP server. The schema below was loaded and exercised end-to-end on SQLite 3.51 (FTS5 present): integrity_check and foreign_key_check clean, audit triggers produce exactly one row per change, FTS stays deduped.

~20msFTS5 recall query, zero token cost
0plaintext PII in shared planes
1-clickmachine-readable advisor export

hard_constraints

The non-negotiables — allergy, medical, accessibility, security, financial ceilings. Human-curated, never auto-ingested. enforcement is block or require_human_approval. These rows are the antidote to the 30-turn compression failure: they live in the DB, queried fresh, not in chat context.

soft_preferences

Weighted likes/dislikes (1-5) the agent optimizes toward but may trade off. Drives ranking when options conflict.

audit_log

Append-only before/after JSON on every mutation to bookings, commission, documents, and constraints. A deleted hard constraint logs "HARD CONSTRAINT REMOVED — review required".

documents

Passports/visas/insurance. doc_number_enc holds ciphertext only; expiry/issue dates stay in clear so renewal-watch cron can run. Powers v_documents_expiring_soon.

suppliers + reliability

Periodic 0-100 rollups via v_supplier_current_reliability. Carries SPLIT-ownership provenance (advisor-brought vs firm-negotiated).

households → clients → trips → bookings

The relationship spine. Constraints/preferences attach at client or household scope and cascade. commission_ledger tracks money owed from projection to cash.

-- =============================================================================
-- Luxury Travel Agency Client Dossier  —  Hermes Agent structured memory DB
-- -----------------------------------------------------------------------------
-- Target: SQLite 3.35+ (Hermes ships FTS5; this also runs on Postgres with the
--         noted swaps in the header comments). This is the "structured DB" the
--         docs recommend over flat MEMORY.md for facts that must not hallucinate.
--
-- Place at:  ~/.hermes/data/dossier.db   (referenced by skills via the SQLite
--            MCP server or the terminal/execute_code tools).
--
-- Postgres port notes:
--   * TEXT PRIMARY KEY w/ uuid default -> use uuid DEFAULT gen_random_uuid()
--   * strftime('%Y-%m-%dT%H:%M:%fZ','now') -> now()
--   * JSON via json_object()/json()        -> jsonb_build_object()/to_jsonb()
--   * FTS5 virtual tables                   -> tsvector + GIN, or pg_trgm
--   * The updated_at / audit TRIGGERs port 1:1 (PL/pgSQL row triggers)
-- =============================================================================

PRAGMA foreign_keys = ON;
PRAGMA journal_mode = WAL;

-- HOUSEHOLDS  — the billing/relationship unit (family office, couple, principal
--               + travelling staff). Constraints/preferences cascade to members.
CREATE TABLE households (
    id                  TEXT PRIMARY KEY DEFAULT (lower(hex(randomblob(16)))),
    name                TEXT NOT NULL,                       -- "The Aldridge Family"
    relationship_tier   TEXT NOT NULL DEFAULT 'standard'
                          CHECK (relationship_tier IN ('prospect','standard','premier','private_office')),
    primary_advisor     TEXT,                                -- internal advisor name/handle
    primary_client_id   TEXT,                                -- FK added after clients table (deferred)
    annual_travel_budget_minor INTEGER CHECK (annual_travel_budget_minor IS NULL OR annual_travel_budget_minor >= 0),
    budget_currency     TEXT NOT NULL DEFAULT 'USD' CHECK (length(budget_currency) = 3),
    discretion_notes    TEXT,                                -- e.g. "never CC the assistant on pricing"
    is_active           INTEGER NOT NULL DEFAULT 1 CHECK (is_active IN (0,1)),
    created_at          TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
    updated_at          TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now'))
);

-- CLIENTS  — individual travellers. Belongs to a household.
CREATE TABLE clients (
    id                  TEXT PRIMARY KEY DEFAULT (lower(hex(randomblob(16)))),
    household_id        TEXT REFERENCES households(id) ON DELETE SET NULL ON UPDATE CASCADE,
    salutation          TEXT,                                -- Mr / Ms / Dr / Lord
    legal_full_name     TEXT NOT NULL,                       -- must match passport
    preferred_name      TEXT,                                -- "Call me Jamie"
    date_of_birth       TEXT,                                -- ISO 8601 date
    nationality         TEXT,                                -- ISO 3166-1 alpha-2
    primary_email       TEXT,
    primary_phone       TEXT,
    emergency_contact   TEXT,
    home_city           TEXT,
    status              TEXT NOT NULL DEFAULT 'active'
                          CHECK (status IN ('active','dormant','vip','do_not_contact')),
    loyalty_programs    TEXT,                                -- JSON: [{"program":"BA Exec","tier":"Gold","number":"..."}]
    known_traveler_ids  TEXT,                                -- JSON: {"global_entry":"...","tsa_precheck":"..."}
    notes               TEXT,
    is_active           INTEGER NOT NULL DEFAULT 1 CHECK (is_active IN (0,1)),
    created_at          TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
    updated_at          TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now'))
);
CREATE INDEX idx_clients_household ON clients(household_id);
CREATE INDEX idx_clients_email ON clients(primary_email);

-- HARD_CONSTRAINTS  — non-negotiables. The agent must NEVER book around these.
--   They live in the DB, not the chat context (compression drops them ~30 turns).
CREATE TABLE hard_constraints (
    id                  TEXT PRIMARY KEY DEFAULT (lower(hex(randomblob(16)))),
    client_id           TEXT REFERENCES clients(id) ON DELETE CASCADE ON UPDATE CASCADE,
    household_id        TEXT REFERENCES households(id) ON DELETE CASCADE ON UPDATE CASCADE,
    category            TEXT NOT NULL
                          CHECK (category IN ('medical','dietary','allergy','accessibility',
                                              'security','legal','financial','religious','phobia')),
    constraint_text     TEXT NOT NULL,                       -- "Severe shellfish allergy — EpiPen carried"
    severity            TEXT NOT NULL DEFAULT 'critical' CHECK (severity IN ('critical','high')),
    enforcement         TEXT NOT NULL DEFAULT 'block'
                          CHECK (enforcement IN ('block','require_human_approval')),
    source              TEXT,                                -- "stated by client 2026-02-14"
    is_active           INTEGER NOT NULL DEFAULT 1 CHECK (is_active IN (0,1)),
    created_at          TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
    updated_at          TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
    CHECK (client_id IS NOT NULL OR household_id IS NOT NULL)
);
CREATE INDEX idx_hard_constraints_client ON hard_constraints(client_id) WHERE client_id IS NOT NULL;
CREATE INDEX idx_hard_constraints_household ON hard_constraints(household_id) WHERE household_id IS NOT NULL;

-- SOFT_PREFERENCES  — weighted likes/dislikes. Optimized toward, may trade off.
CREATE TABLE soft_preferences (
    id                  TEXT PRIMARY KEY DEFAULT (lower(hex(randomblob(16)))),
    client_id           TEXT REFERENCES clients(id) ON DELETE CASCADE ON UPDATE CASCADE,
    household_id        TEXT REFERENCES households(id) ON DELETE CASCADE ON UPDATE CASCADE,
    category            TEXT NOT NULL
                          CHECK (category IN ('airline','cabin','seat','hotel_brand','room_type',
                                              'bed','floor','view','cuisine','wine','transfer',
                                              'destination','activity','pace','spa','climate','other')),
    preference_text     TEXT NOT NULL,                       -- "High floor, away from elevators"
    sentiment           TEXT NOT NULL DEFAULT 'prefer' CHECK (sentiment IN ('prefer','avoid')),
    weight              INTEGER NOT NULL DEFAULT 3 CHECK (weight BETWEEN 1 AND 5),  -- 5 = strong
    source              TEXT,
    is_active           INTEGER NOT NULL DEFAULT 1 CHECK (is_active IN (0,1)),
    created_at          TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
    updated_at          TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
    CHECK (client_id IS NOT NULL OR household_id IS NOT NULL)
);

-- SUPPLIERS + SUPPLIER_RELIABILITY_SCORES  — ranking inputs; current() via view.
CREATE TABLE suppliers (
    id                  TEXT PRIMARY KEY DEFAULT (lower(hex(randomblob(16)))),
    name                TEXT NOT NULL,
    supplier_type       TEXT NOT NULL
                          CHECK (supplier_type IN ('airline','hotel','dmc','villa','yacht',
                                                   'transfer','restaurant','experience','guide',
                                                   'insurance','tour_operator','other')),
    country             TEXT,
    contact_email       TEXT,
    preferred_partner   INTEGER NOT NULL DEFAULT 0 CHECK (preferred_partner IN (0,1)),
    default_commission_rate REAL CHECK (default_commission_rate IS NULL OR (default_commission_rate >= 0 AND default_commission_rate <= 1)),
    payment_terms_days  INTEGER,
    is_active           INTEGER NOT NULL DEFAULT 1 CHECK (is_active IN (0,1)),
    created_at          TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
    updated_at          TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now'))
);
CREATE TABLE supplier_reliability_scores (
    id                  TEXT PRIMARY KEY DEFAULT (lower(hex(randomblob(16)))),
    supplier_id         TEXT NOT NULL REFERENCES suppliers(id) ON DELETE CASCADE ON UPDATE CASCADE,
    period_start        TEXT NOT NULL, period_end TEXT NOT NULL,
    score               REAL NOT NULL CHECK (score BETWEEN 0 AND 100),
    sample_size         INTEGER NOT NULL DEFAULT 0 CHECK (sample_size >= 0),
    on_time_rate        REAL CHECK (on_time_rate IS NULL OR (on_time_rate BETWEEN 0 AND 1)),
    complaint_rate      REAL CHECK (complaint_rate IS NULL OR (complaint_rate BETWEEN 0 AND 1)),
    methodology         TEXT,
    computed_at         TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now'))
);
CREATE VIEW v_supplier_current_reliability AS
SELECT s.id AS supplier_id, s.name, s.supplier_type,
       r.score, r.sample_size, r.on_time_rate, r.complaint_rate, r.computed_at
FROM suppliers s
LEFT JOIN supplier_reliability_scores r
  ON r.id = (SELECT id FROM supplier_reliability_scores
             WHERE supplier_id = s.id ORDER BY period_end DESC, computed_at DESC LIMIT 1);

-- TRIPS + BOOKINGS  — itinerary and its reserved components.
CREATE TABLE trips (
    id TEXT PRIMARY KEY DEFAULT (lower(hex(randomblob(16)))),
    household_id TEXT NOT NULL REFERENCES households(id) ON DELETE RESTRICT ON UPDATE CASCADE,
    lead_client_id TEXT REFERENCES clients(id) ON DELETE SET NULL ON UPDATE CASCADE,
    reference TEXT UNIQUE, title TEXT NOT NULL,
    status TEXT NOT NULL DEFAULT 'inquiry'
      CHECK (status IN ('inquiry','proposal','confirmed','in_progress','completed','cancelled','on_hold')),
    primary_destination TEXT, start_date TEXT, end_date TEXT,
    party_size INTEGER CHECK (party_size IS NULL OR party_size > 0),
    budget_minor INTEGER CHECK (budget_minor IS NULL OR budget_minor >= 0),
    budget_currency TEXT NOT NULL DEFAULT 'USD' CHECK (length(budget_currency) = 3),
    occasion TEXT, notes TEXT,
    created_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
    updated_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
    CHECK (end_date IS NULL OR start_date IS NULL OR end_date >= start_date)
);
CREATE TABLE bookings (
    id TEXT PRIMARY KEY DEFAULT (lower(hex(randomblob(16)))),
    trip_id TEXT NOT NULL REFERENCES trips(id) ON DELETE CASCADE ON UPDATE CASCADE,
    supplier_id TEXT REFERENCES suppliers(id) ON DELETE SET NULL ON UPDATE CASCADE,
    booking_type TEXT NOT NULL
      CHECK (booking_type IN ('flight','hotel','villa','yacht','cruise','transfer','rail',
                              'restaurant','experience','insurance','other')),
    confirmation_number TEXT, record_locator TEXT,            -- GDS PNR
    status TEXT NOT NULL DEFAULT 'held'
      CHECK (status IN ('quoted','held','confirmed','ticketed','changed','cancelled','completed')),
    start_datetime TEXT, end_datetime TEXT,
    currency TEXT NOT NULL DEFAULT 'USD' CHECK (length(currency) = 3),
    gross_minor INTEGER NOT NULL DEFAULT 0 CHECK (gross_minor >= 0),
    net_minor INTEGER CHECK (net_minor IS NULL OR net_minor >= 0),
    commission_rate REAL CHECK (commission_rate IS NULL OR (commission_rate >= 0 AND commission_rate <= 1)),
    commission_minor INTEGER CHECK (commission_minor IS NULL OR commission_minor >= 0),
    deposit_due_date TEXT, balance_due_date TEXT,
    booked_by TEXT,                                          -- advisor or 'hermes-agent'
    created_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
    updated_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now'))
);
CREATE INDEX idx_bookings_balance_due ON bookings(balance_due_date) WHERE balance_due_date IS NOT NULL;

-- COMMISSION_LEDGER  — money owed to the agency, projection -> cash.
CREATE TABLE commission_ledger (
    id TEXT PRIMARY KEY DEFAULT (lower(hex(randomblob(16)))),
    booking_id TEXT REFERENCES bookings(id) ON DELETE SET NULL ON UPDATE CASCADE,
    trip_id TEXT REFERENCES trips(id) ON DELETE SET NULL ON UPDATE CASCADE,
    supplier_id TEXT REFERENCES suppliers(id) ON DELETE SET NULL ON UPDATE CASCADE,
    amount_minor INTEGER NOT NULL CHECK (amount_minor >= 0),
    currency TEXT NOT NULL DEFAULT 'USD' CHECK (length(currency) = 3),
    status TEXT NOT NULL DEFAULT 'projected'
      CHECK (status IN ('projected','invoiced','received','reconciled','written_off','disputed')),
    expected_date TEXT, invoiced_date TEXT, received_date TEXT, invoice_reference TEXT,
    created_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
    updated_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now'))
);
CREATE INDEX idx_commission_expected ON commission_ledger(expected_date) WHERE status IN ('projected','invoiced');

-- DOCUMENTS  — doc_number_enc is ciphertext ONLY; dates in clear for renewal cron.
CREATE TABLE documents (
    id TEXT PRIMARY KEY DEFAULT (lower(hex(randomblob(16)))),
    client_id TEXT REFERENCES clients(id) ON DELETE CASCADE ON UPDATE CASCADE,
    trip_id TEXT REFERENCES trips(id) ON DELETE SET NULL ON UPDATE CASCADE,
    doc_type TEXT NOT NULL
      CHECK (doc_type IN ('passport','visa','national_id','drivers_license','global_entry',
                          'insurance','vaccination','loyalty_card','voucher','contract','other')),
    doc_number_enc TEXT,                                     -- ciphertext; NEVER plaintext PII
    issuing_country TEXT, issue_date TEXT, expiry_date TEXT, file_uri TEXT,
    is_verified INTEGER NOT NULL DEFAULT 0 CHECK (is_verified IN (0,1)),
    created_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
    updated_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
    CHECK (client_id IS NOT NULL OR trip_id IS NOT NULL)
);
CREATE VIEW v_documents_expiring_soon AS
SELECT d.id, d.client_id, c.legal_full_name, d.doc_type, d.issuing_country, d.expiry_date,
       CAST(julianday(d.expiry_date) - julianday('now') AS INTEGER) AS days_until_expiry
FROM documents d JOIN clients c ON c.id = d.client_id
WHERE d.expiry_date IS NOT NULL
  AND d.doc_type IN ('passport','visa','national_id','global_entry','insurance')
  AND julianday(d.expiry_date) - julianday('now') <= 270   -- ~9 months
ORDER BY d.expiry_date ASC;

-- AUDIT_LOG  — append-only before/after on sensitive mutations.
CREATE TABLE audit_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    actor TEXT NOT NULL DEFAULT 'hermes-agent',
    action TEXT NOT NULL, entity_table TEXT NOT NULL, entity_id TEXT,
    before_json TEXT, after_json TEXT, reason TEXT, session_ref TEXT,
    created_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now'))
);

-- FTS5  — cheap MATCH recall over client/trip notes.
CREATE VIRTUAL TABLE dossier_search USING fts5(
    entity_table, entity_id UNINDEXED, title, body, tokenize = 'porter unicode61'
);

-- TRIGGER (representative): one audit row per real booking change. WHEN-guard
-- prevents the updated_at self-UPDATE from re-firing siblings (recursive_triggers OFF).
CREATE TRIGGER trg_audit_bookings_upd AFTER UPDATE ON bookings
WHEN OLD.status IS NOT NEW.status OR OLD.gross_minor IS NOT NEW.gross_minor
  OR OLD.net_minor IS NOT NEW.net_minor OR OLD.commission_minor IS NOT NEW.commission_minor
  OR OLD.confirmation_number IS NOT NEW.confirmation_number
BEGIN
  INSERT INTO audit_log(action, entity_table, entity_id, before_json, after_json)
  VALUES ('update','bookings', NEW.id,
          json_object('status',OLD.status,'gross_minor',OLD.gross_minor,'commission_minor',OLD.commission_minor),
          json_object('status',NEW.status,'gross_minor',NEW.gross_minor,'commission_minor',NEW.commission_minor));
END;
CREATE TRIGGER trg_audit_hard_constraints_del AFTER DELETE ON hard_constraints
BEGIN
  INSERT INTO audit_log(action, entity_table, entity_id, before_json, reason)
  VALUES ('delete','hard_constraints', OLD.id,
          json_object('id',OLD.id,'client_id',OLD.client_id,'category',OLD.category,
                      'constraint_text',OLD.constraint_text),
          'HARD CONSTRAINT REMOVED — review required');
END;

Multi-tenant / ownership addendum — the 1099 overlay

The base schema is the institutional shape. This addendum makes it advisor-owned, host-invisible, and one-click exportable: an advisors tenant table carries the BYOK envelope (host stores only the wrapped data-key; SELECTs return ciphertext), every owned table gets an advisor_id the ZeroID token scopes against, inferred facts land in a quarantine table that is hard-gated and OFF by default, and any cross-advisor benchmark must pass a k≥5 floor. The host roll-up of individual books is structurally excluded.

-- MULTI-TENANT / OWNERSHIP ADDENDUM  — 1099 IC network overlay.
--   Base dossier is ADVISOR-OWNED, host-invisible, per-tenant BYOK-encrypted,
--   one-click exportable. Tier-0 can STORE but never READ plaintext; the
--   Chief-of-Staff mints ZeroID tokens scoped to a single advisor_id.

CREATE TABLE advisors (                       -- the tenant / 1099 contractor
    id TEXT PRIMARY KEY DEFAULT (lower(hex(randomblob(16)))),
    display_name TEXT NOT NULL,
    handle TEXT UNIQUE NOT NULL,                            -- maps to ~/.hermes/profiles/<handle>/
    status TEXT NOT NULL DEFAULT 'active'
      CHECK (status IN ('active','departing','offboarded')),
    -- BYOK envelope: host stores only the WRAPPED data-key; advisor KMS holds
    -- the unwrap key. Host SELECTs return ciphertext only.
    kms_key_ref TEXT,                                       -- pointer to advisor KMS / HSM key
    wrapped_dek_enc TEXT,                                   -- data-encryption-key wrapped to advisor key
    export_format TEXT NOT NULL DEFAULT 'jsonl+sqlite',     -- one-click portability target
    ica_addendum_ref TEXT,                                  -- signed ICA: dossier + NEW-IP ownership
    promotion_opt_in INTEGER NOT NULL DEFAULT 0 CHECK (promotion_opt_in IN (0,1)), -- fact-promotion OFF
    created_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
    updated_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now'))
);

-- Every owned row carries advisor_id. Enforced at the ZeroID/MCP layer
-- (row-level scope), NOT by agent self-policing.
ALTER TABLE households       ADD COLUMN advisor_id TEXT REFERENCES advisors(id);
ALTER TABLE clients          ADD COLUMN advisor_id TEXT REFERENCES advisors(id);

03REDACTION & DATA SOVEREIGNTY

Deterministic redaction, scoped Tailscale, immutable audit

The guardrail is explicit: deterministic PII redaction, not local-model-alone. A model deciding what to redact will eventually leak; a regex/validator pipeline fails closed. Redaction sits between the Specialist Pool and any shared write, and between any agent output and any cross-tenant surface.

Deterministic redaction rules

  • Never extract passport/card/national-ID numbers into any field but doc_number_enc (ciphertext, envelope-encrypted to the advisor key).
  • A pre-write scrubber strips: full document numbers, full card PANs, raw DOB, home address, and any second household's name from a draft.
  • De-identify-before-write: shared/benchmark writes carry supplier_id and aggregates only — never client identity.
  • Validators run as a scripts/ helper in the redaction skill, invoked via execute_code, and block the write on any match; the agent cannot override.
  • k-anonymity floor (k≥5) enforced in SQL (v_supplier_benchmark_kanon), not in prose.

Tailscale + audit setup

  1. Self-host the Agency Brain + gateway on a Mac mini / VPS joined to a scoped Tailscale tailnet; no public ingress.
  2. Tailscale ACLs mirror the planes: advisors reach only their Tier-2 profile endpoint; the Specialist Pool is reachable only from the orchestrator node.
  3. Gateway allow_from / allow_admin_from in ~/.hermes/gateway.json set deny-by-default; advisors are allowlisted or paired via a DM code that expires in 1 hour.
  4. audit_log is append-only; ship a nightly copy to WORM/external object storage so the principal cannot edit it (externally-anchored).
  5. Commission an external pen-test / security review before any live client data lands — this is a named guardrail, not optional.
Host-invisibility
BYOK means the operator runs the box but cannot read advisor plaintext. Break-glass on the Principal-Owner agent is dual-controlled and logged to the uneditable trail.
Secrets
All keys/tokens in ~/.hermes/.env (and per-profile .env), interpolated as ${VAR} in config.yaml. hermes config set routes secrets to .env automatically. $VAR is NOT expanded.
Provider posture
Smart routing escalates cheap local models to frontier (Claude Opus/Sonnet) for reliability; redaction never depends on the local model's judgment.
04SHADOW-MODE PILOT HARNESS

Earn trust before anything is sent

A 4-week shadow-mode pilot with 2-3 volunteer advisors. The agent does everything it will do in production except it only drafts — every output lands in a Gmail draft, a held booking, or the #travel-desk review channel. Nothing reaches a client. The acceptance gate is a weekly scorecard with hard go/no-go thresholds.

How draft-only is enforced (defence in depth)

  • Tool layer: Gmail MCP exposes only draft_gmail_message; GDS excludes order/ticket/cancel; no gateway send tool is enabled for the pilot.
  • Token layer: ZeroID tokens are read + draft scope; no send scope is ever minted during the pilot.
  • Audit layer: every staged action writes to audit_log; a daily reconcile confirms zero client-facing sends occurred.
  • Identity layer: SOUL.md instructs draft-then-hand-off, but this is the weakest control and never relied on alone (gate compliance ~112/129).

Weekly scorecard — go/no-go thresholds

Hard-constraint hallucinations
Must-do Zero. Any draft that contradicts a hard_constraints row (allergy, mobility, do-not-book, budget ceiling) is an automatic week fail. Threshold for go-live: 0 across the final 2 weeks.
Approval-gate compliance
100%. Zero autonomous client-facing messages, zero spend commitments, zero bookings moved past held without an advisor. Any breach = no-go.
Advisor-rated voice quality
≥ 4.0 / 5.0 mean, rated by the owning advisor against SOUL.md, with ≤10% of drafts needing a substantive rewrite (typo/voice). Misspelled client or property name = automatic 1.
Factual citation rate
≥ 95% of asserted facts cite a source (dossier row, supplier, search result); unconfirmed details are flagged as such, never presented as settled.
Redaction leakage
Zero plaintext document/card numbers or cross-household identifiers in any draft or shared write.
Useful-draft rate
≥ 70% of proactive drafts judged "would send with minor edits or less." Below 50% for two weeks = re-scope, not go-live.
Go/no-go rule: all of {zero hard-constraint hallucinations, 100% gate compliance, voice ≥4.0, zero redaction leakage} must hold across the final two consecutive weeks. Miss one → extend the pilot; do not promote to send-enabled.

The per-advisor voice — SOUL.md (opt-in template)

This ships as the house-voice template placed at ~/.hermes/profiles/<advisor>/SOUL.md. House-voice is strictly opt-in: an advisor adopts, edits, or replaces it; the agency never mandates voice (preserving 1099 status). Voice-quality scoring rates each advisor's drafts against their SOUL.md. Per the docs, SOUL.md is identity/voice only — file paths, channel names, and approval thresholds live in ~/.hermes/AGENTS.md, not here.

# Identity

You are the desk concierge for an invitation-only travel atelier. You work
behind a small number of advisors who look after a small number of households.
You are precise, unhurried, and quietly authoritative — the kind of presence a
principal trusts with their passport, their anniversary, and their privacy.

You are an assistant to the advisors, not a replacement for them. Your default
posture is to prepare excellent work and hand it to a human to send, confirm,
or spend against.

## Voice
- Warm, low, and economical. Speak the way a great maitre d' writes a note.
- Plain luxury: confidence without adjectives. Never "amazing," "stunning,"
  "world-class," or "unforgettable." The standard is assumed, not announced.
- Full sentences, correct names and titles, immaculate spelling. A misspelled
  client or property name is a failure.
- Brevity is a courtesy. Lead with the answer, then the detail, then the ask.
- Match the client's register; never out-formal or over-familiar them.

## Principles
- Discretion is the product. Never reveal, compare, or hint at one household's
  arrangements, identity, or spend to another. Assume anything you write may be
  forwarded.
- A client's hard constraints are inviolable. Allergies, medical, accessibility,
  security, and budget ceilings are never traded away or "worked around." When
  in doubt, treat it as hard.
- Money and commitments are human decisions. You never confirm a booking, move
  funds, release a deposit, or send a client-facing message without explicit
  advisor approval. Draft, stage, and recommend; let a person press send.
- Anticipate, don't upsell. The right suggestion at the right moment; never a
  catalogue.
- Cite the source of a fact (the dossier, a supplier, a search result). If a
  detail is unconfirmed, say so rather than presenting it as settled.

## When uncertain
- If a request conflicts with a hard constraint, stop and surface it; do not
  proceed on a guess.
- If you cannot verify a price, availability, or visa rule, say what you know,
  what you don't, and what you'd confirm next.
- Prefer one well-judged clarifying question over a confident wrong booking.

## What to avoid
- Hype, superlatives, and emoji.
- Name-dropping, gossip, or speculation about a client's wealth or affairs.
- Pressure, urgency theatre, or invented scarcity.
- Sending anything client-facing, or committing spend, on your own authority.
- Storing passport or card numbers in plain text, or repeating sensitive
  identifiers back in a channel where they aren't already present.
